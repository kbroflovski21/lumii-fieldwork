状态：v2 设计定稿
日期：2026-05-15
仓库：`aro-network/lumii-fieldwork`
视觉规范：`docs/global-ui-guidance.md`

## 1. 页面定位

服务主管页面是一个**以 AI 对话为主要交互手段的文档管理界面**。

服务主管通过这个界面管理两类服务规范：

- **通用规范**：某一业务大类中，所有上门服务都必须遵循的流程性规则。例如"国家长期护理保险"要求所有上门服务人员必须自报家门、不得推销、必须询问满意度。这些要求与具体服务项目无关，是通用的流程约束。
- **服务项目规范**：实施某一项具体服务时，必须遵循的专业操作步骤。例如"清洁照护-口腔清洁"的 SOP 定义了从准备物品到记录口腔状况的 8 个操作步骤。

这两类规范作用在正交的两个维度上：通用规范是水平的流程层，服务项目规范是垂直的专业层。社工在执行一次上门服务时，需要同时遵守适用的通用规范和具体项目规范。

## 2. 规范的内部结构

每一个规范（无论通用还是服务项目）都包含固定的三个文件：

| 文件 | 性质 | 产生方式 |
|------|------|---------|
| **SOP** | 规章或标准定义的操作要求，是规范的源头 | 服务主管手动录入、上传文档、或口述后由 AI 整理 |
| **服务中实时督导要求** | 定义 AI 在社工上门服务过程中如何做语音督导 | AI 基于 SOP 自动推理生成。服务主管可审核和修改 |
| **服务后报告要求** | 定义服务结束后报告中需要提取和总结哪些信息 | AI 基于 SOP 自动推理生成。服务主管可审核和修改 |

**核心产品逻辑**：SOP 是唯一的源头。当 SOP 被创建或修改时，AI 自动理解 SOP 的要求，推理得出"服务过程中应该怎么督导"和"服务后报告应该包含什么"。服务主管不需要从零写督导和报告要求，只需要审核 AI 的推理结果并做微调。

## 3. 版本管理

每个规范文件夹有一个版本号。任何文件（SOP、督导要求、报告要求）的更新都会导致版本号 +1。

版本历史记录每次变更的：版本号、日期、变更摘要。

用户可以通过以下方式访问版本信息：
- 在 Tab 中，规范文件夹名称旁显示当前版本号（如 v3）
- 展开文件夹后，点击「v3 版本历史」按钮，在聊天框查看完整版本记录
- 在聊天中询问 AI，如"口腔清洁的版本历史是什么"、"对比 v2 和 v3 有什么变化"

## 5. 交互模型

### 5.1 双路径操作

服务主管可以通过两条独立路径完成操作：

**路径 A：直接操作（目录 + 文档 View）**
- 重命名：三点菜单 → 内联编辑（直接在目录树中改名）→ 回车保存
- 删除文件夹：三点菜单 → 浏览器 confirm 对话框 → 确认后直接删除
- 删除文档：文档 header「删除文档」按钮 → confirm 对话框 → 确认后文档置空
- 编辑文档：文档 header「编辑」按钮 → textarea 直接编辑 → 保存（版本 +1）
- 查看版本历史：文档 header「v3 历史」按钮 → 文档区域切换到版本列表视图

**路径 B：AI 对话（右侧聊天面板）**
- 以上所有操作也可以通过自然语言对话完成
- AI 可以执行结构化操作（创建/更新/删除），通过 JSON action 驱动状态变更
- 适合复杂操作：上传文档解析、批量更新、基于 SOP 推理生成督导和报告要求

**添加规范**：只通过 AI 对话路径。因为新增规范涉及 SOP 解析和 AI 推理（自动生成督导要求和报告要求），需要 AI 参与。入口：目录底部的「+ 添加规范」按钮触发 AI 对话。

### 5.2 三面板布局

三个面板可独立收起和展开：

- **收起**：面板顶部方向箭头按钮，至少保留一个面板
- **收起后**：显示 32px 竖向标签条（写有面板名称），点击恢复
- **面板宽度可拖动**：目录和文档之间、文档和对话之间有 4px 拖动条
- 约束：目录 min 180px / max 400px，对话 min 280px / max 600px

### 5.3 版本历史视图

点击文档 header 的「v3 历史」按钮 → 文档区域切换到版本历史视图：

- 顶部紫色背景条：文档标题 + 当前查看的版本号
- 「← 回到最新版本 (v3)」按钮
- 版本条：水平排列的版本按钮（v1 · 日期、v2 · 日期...），可横向滚动
- 点击版本按钮 → 下方显示该版本内容（半透明，表示非当前版本）
- 版本管理粒度到具体文档（SOP、督导要求、报告要求各自独立版本）

### 5.4 AI 对话能力

对话由 Qwen3-max 驱动。AI 了解当前文档上下文（用户正在查看哪个规范的哪个文件）。

支持的对话操作：
- 查看：「查看口腔清洁的 SOP」
- 编辑：「帮我改一下这个文档的第三条」（AI 知道"这个文档"是什么）
- 删除：「我想删除口腔清洁的规范」→ AI 确认 → 用户确认 → 执行
- 新增：「新增一个排泄照护-协助排泄的规范」→ AI 引导提供 SOP → AI 推理生成三个文件
- 上传解析：上传文件 → AI 解析内容 → 自动归类创建/更新规范
- 版本查询：「这个文档上一个版本改了什么」

### 5.5 二次确认规则

所有破坏性操作使用自定义确认弹窗（非浏览器 confirm），弹窗包含标题、描述、取消和确认按钮。

| 操作 | 直接操作路径 | AI 对话路径 |
|------|------------|------------|
| 重命名 | 内联编辑，回车即生效 | AI 执行，无需额外确认 |
| 删除文件夹 | 自定义确认弹窗：「所有文件将全部删除，不可撤销」| AI 回复确认提示 + 确认/取消按钮（一次性） |
| 删除 SOP 文档 | 自定义确认弹窗：「删除后督导要求和报告要求将无法对应到 SOP」| 同上 |
| 删除督导/报告文档 | 自定义确认弹窗：「删除后可通过 AI 基于 SOP 重新生成」| 同上 |
| 编辑保存 | 点击保存即生效 | AI 预览修改 + 确认/取消按钮（一次性） |
| 新增规范 | N/A（走 AI 路径） | AI 预览 + 确认/取消按钮（一次性） |

确认弹窗设计：
- 半透明遮罩 + 居中白色卡片（400px，12px 圆角）
- 标题（15px 加粗）+ 描述文字（13px）
- 底部右对齐：灰色「取消」+ 红色「确认删除」
- 点击遮罩可取消

AI 对话中的确认/取消按钮为一次性使用，点击后消失，不可重复操作。

## 6. Tab 面板

### Tab 栏

两个 Tab：「通用规范」「服务项目规范」。选中态蓝色下划线 + 加粗，未选中灰色。

### 规范文件夹列表

每个规范显示为一个可展开的文件夹卡片：

**收起状态**：
- 📁 图标 + 规范名称 + 版本号标签（紫色背景，如 v3）
- 三文件状态：SOP ✓ | 督导 ✓ | 报告 ✓（绿色标签=完整，黄色=不完整，灰色横杠=空）
- 右侧展开箭头

**展开状态**：
- 📂 图标（打开态）
- 三个文件卡片（堆叠排列）：
  - 文件名 + 来源标签（手动录入 or 「AI 生成」紫色标签）
  - 内容预览（前两行，灰色小字）
  - 「查看完整内容 →」链接（点击在聊天框推送全文）
- 操作按钮行：
  - 「v3 版本历史」（紫色描边）——点击在聊天框推送版本记录
  - 「编辑」（蓝色描边）——点击等同于用户发送"我想编辑 XXX"
  - 「删除」（红色描边）——点击等同于用户发送"我想删除 XXX"

### 添加按钮

每个 Tab 列表最下方有「+ 添加新的 XX 规范」按钮（虚线框样式）。点击等同于用户发送"我想添加新的 XX 规范"，触发 AI 对话流程。

### 文件夹三点菜单

鼠标悬停在文件夹行上时，右侧出现 ⋮ 三点图标。点击弹出下拉菜单：
- **重命名**：点击后文件夹名变为内联 input，回车保存，Esc 取消
- **删除**：点击后弹出确认弹窗，确认后删除整个文件夹

菜单为浮层，点击菜单外任意位置自动关闭。

### 文档操作按钮

文档 header 右侧按钮组（非编辑状态）：
- 「v3 历史」（紫色描边）→ 文档区域切换到版本历史视图
- 「编辑」（蓝色描边）→ 文档内容变为 textarea 可编辑
- 「删除文档」（红色描边）→ 弹出确认弹窗（SOP 和非 SOP 提示文案不同）

文档 header 右侧按钮组（编辑状态）：
- 「保存」（蓝色实底）→ 保存并退出编辑，文档版本 +1
- 「取消」（灰色）→ 撤销修改，退出编辑

## 8. 数据模型

```typescript
interface StandardFolder {
  id: string;
  type: 'general' | 'service';
  name: string;
  version: number;
  sop: StandardDocument | null;
  supervision: StandardDocument | null;
  report: StandardDocument | null;
  history: VersionRecord[];
}

interface StandardDocument {
  status: 'complete' | 'incomplete' | 'empty';
  content: string;
  source: 'manual' | 'ai_generated';
}

interface VersionRecord {
  version: number;
  date: string;
  summary: string;
}
```

## 9. 与社工上门服务的关系

产品设计中，社工在执行一次上门服务时：

1. 系统加载该服务适用的**通用规范**的督导要求 + 报告要求
2. 系统加载该服务对应的**服务项目规范**的督导要求 + 报告要求
3. 两者合并后注入 LLM prompt，驱动实时语音督导和事后报告生成

当服务主管在本页面修改了任何规范，下一次社工执行相关服务时，督导行为和报告内容会自动反映最新的规范版本。

## 10. 技术实现要点

### LLM 对话端点

`POST /api/supervisor-chat`

- 输入：用户消息 + 当前所有规范文件夹状态 + 最近 10 条对话历史
- 系统 prompt 包含完整的角色定义、文档结构说明、可用操作列表和确认规则
- 强制 JSON 输出（`response_format: { type: 'json_object' }`）
- 输出：`{ reply: string, actions: Action[] }`
- Action 类型：`create_folder`、`update_doc`、`delete_folder`
- 前端接收到 actions 后立即更新本地文件夹状态

### 文件上传

- 输入框左侧📎按钮，触发 `<input type="file">`
- 客户端读取文件内容（`file.text()`），截取前 8000 字
- 作为用户消息发送给 LLM：「我上传了文件 XXX，内容如下：...」
- LLM 解析后在 actions 中返回创建/更新操作
