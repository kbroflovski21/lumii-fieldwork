状态：设计定稿

仓库：`aro-network/lumii-fieldwork`

## 1. 核心定位

服务主管页面本质上是一个**文档管理界面**。

服务主管的核心职能是管理两类规范：
- **通用规范**：某一大类业务中，所有上门服务都需要遵循的流程性规则
- **服务项目规范**：实施某一项具体服务时，需要遵循的专业步骤性规则

这两类规范作用在两个不同维度：
- 通用规范 = 流程性要求（无论做什么服务都要遵守）
- 服务项目规范 = 专业步骤性要求（做特定服务时的操作标准）

服务主管定义这些规范的根本目的是：基于 SOP，进一步驱动「服务后报告要求」和「服务中实时督导要求」。

## 2. Tab 结构

| Tab | 内容 |

| 服务项目规范 | 按具体服务项目组织的专业操作规范 |

### 固定三文件结构

每个最末级文件夹（无论通用规范还是服务项目规范）都包含固定的三个文件：

| 文件 | 定义 | 由谁产生 |
|------|------|---------|
| **SOP** | 规章或标准定义的操作要求 | 服务主管录入或上传 |
| **服务中实时督导要求** | 上门服务中 AI 语音督导如何介入 | AI 基于 SOP 推理生成，服务主管可审核修改 |
| **服务后报告要求** | 服务结束后报告中需要总结哪些信息 | AI 基于 SOP 推理生成，服务主管可审核修改 |

**关键逻辑**：SOP 是源头。AI 读取 SOP 后，自动推理生成督导要求和报告要求。服务主管查看后可以微调细节。

### 5.4 文件卡片交互

每个文件卡片支持：
- **查看**：「查看完整内容 →」在聊天框推送完整 Markdown 内容
- **编辑**：通过聊天框对话编辑（如「修改口腔清洁的 SOP 第三步」）
- **AI 标签**：「AI 生成」标签表示该文件由 AI 基于 SOP 推理生成，服务主管可审核和修改

### 5.5 对话能力

服务主管可以通过聊天完成以下操作：

**查看**：
- 「查看口腔清洁的 SOP」→ 在聊天框推送 SOP 全文
- 「查看国家长护险的督导要求」→ 推送督导要求全文

**编辑**：
- 「修改口腔清洁的 SOP，增加一步漱口要求」→ AI 更新 SOP → 自动重新推理督导和报告要求
- 「口腔清洁的督导要求里，增加一条：如果社工没有戴手套就提醒」→ 直接编辑督导要求

**批量更新**：
- 「我要更新一些服务的需求」→ AI 让服务主管上传文件或口述
- 服务主管上传 Word → AI 解析 → 自动归类更新对应文件夹中的文件

**AI 推理**：
- 新建 SOP 后，AI 自动基于 SOP 内容推理生成「督导要求」和「报告要求」
- 修改 SOP 后，AI 提示「SOP 已更新，是否需要重新生成督导和报告要求？」

## 6. Demo 范围

Demo 中不需要实现完整的 36 项服务。只需展示：

### 通用规范

**国家长期护理保险**（1 个二级文件夹，3 个文件）

SOP 内容（写死）：
1. 上门服务人员必须在开始服务时自报家门（姓名、所属机构），并确认被服务人员身份
2. 服务过程中不得向被服务人员推销任何商业产品、保健品、保险或理财
3. 不得私下收取费用或接受好处
4. 不得拍摄或传播被服务人员的照片、视频或个人信息
5. 服务结束前必须明确复述本次完成的服务内容，要求被服务人员确认，并询问满意度
6. 服务全程需保持录音，录音自动存档

服务中实时督导要求（AI 基于 SOP 推理，写死）：
1. 开场 1 分钟内未做自报家门和身份确认 → 语音提示一次
2. 出现服务结束迹象但未复述服务内容和询问满意度 → 语音提示一次
3. 检测到推销商品、私下收费等违规行为 → 具体描述违规并提示，重复违规则再次提示

服务后报告要求（AI 基于 SOP 推理，写死）：
- 🟢 语音全文日志存档（默认开启）
1. 是否做了开场确认（自报家门 + 身份确认），如有则记录社工姓名、机构、被服务人员姓名；如无则明确提示缺失
2. 是否询问了被服务人员身体情况，并总结结果
3. 是否进行了预定的健康监测，完成了哪些项目和结果
4. 是否在结束前复述了服务内容并询问满意度
5. 如果询问了满意度，总结被服务人员表达的信息
6. 是否出现任何违规行为，如有则具体描述
7. 服务全程的日期、起止时间

### 服务项目规范

Demo 中展示 1-2 个具体项目即可，例如：

**清洁照护 — 口腔清洁**（作为一个完整示例）

SOP 内容（写死）：
1. 准备物品：软毛牙刷、牙膏、漱口杯、温水、毛巾、弯盘
2. 协助被服务人员取坐位或侧卧位
3. 检查口腔有无溃疡、出血、假牙等情况
4. 挤适量牙膏，用温水浸湿牙刷
5. 按顺序刷牙：先上后下、先外后内、牙面咬合面
6. 协助漱口，观察有无出血或不适
7. 清洁用具，协助被服务人员恢复舒适体位
8. 记录口腔状况及操作过程

**基础健康观察 — 生命体征测量**（作为第二个示例）

SOP 内容（写死）：
1. 确认测量项目（血压、体温、脉搏、呼吸）
2. 准备测量设备，检查设备正常
3. 协助被服务人员取合适体位
4. 按规范操作测量
5. 记录测量数值
6. 与以往数值对比，发现异常及时报告
7. 整理设备

这两个服务项目的督导要求和报告要求由 AI 推理生成（demo 中写死）。

## 7. 数据模型

### 文件夹结构

```typescript
interface StandardFolder {
  id: string;
  type: 'general' | 'service';      // 通用规范 or 服务项目规范
  name: string;                       // 文件夹名（服务项目用"分类-项目"格式，如"清洁照护-口腔清洁"）
  sop: StandardDocument | null;
  supervision: StandardDocument | null;
  report: StandardDocument | null;
}

interface StandardDocument {
  status: 'complete' | 'incomplete' | 'empty';
  content: string;                    // Markdown 内容
  source: 'manual' | 'ai_generated'; // 手动创建 or AI 推理
  lastUpdated: string;
}
```

## 8. 与社工页面的关系

在 demo 中，社工页面的督导行为和报告生成逻辑仍然写死（不从 shared-store 动态读取）。但概念上，社工在执行某次上门服务时：

1. 系统加载该服务适用的**通用规范**（如国家长护险）的督导要求 + 报告要求
2. 系统加载该服务对应的**服务项目规范**（如口腔清洁）的督导要求 + 报告要求
3. 两者合并后驱动实时督导和事后报告

这个合并逻辑在 demo 中不实现，但产品设计中是核心。
