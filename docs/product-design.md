## 1. 产品定位

社区居家养老 AI 督导是一个面向居家养老上门服务场景的 H5 MVP。产品用手机 H5 + 蓝牙耳机先跑通完整业务闭环，未来可平滑迁移到硬件工牌，但不等待硬件。

这里的 MVP 不是演示原型，而是可以直接部署到现场试运行的最小生产系统。它必须覆盖从后台初始化、SOP 输入调试、排班派单、社工上门服务、AI 督导、日志/报告生成、管理审核到家属触达的全链路。

MVP 的核心目标是：

- 社工带手机和蓝牙耳机上门服务，AI 在服务过程中通过语音实时督导。
- 服务结束后，系统自动生成两份核心交付物：内部服务日志和家属健康报告。
- 站长可以排班、实时查看服务状态、通过 Agent 询问异常和服务质量。
- 家属可以在服务后通过飞书 channel 收到报告、查看历史趋势、给服务团队留言。

产品不是“表单录入工具”，而是三类 Agent 协同的服务监督系统：社工端是随身督导，管理端是智能助理，家属端是健康顾问。

## 2. 设计原则

### 2.1 语音优先，服务中零操作

社工在服务过程中不应频繁看手机或操作表单。手机可以放在口袋里，蓝牙耳机承载主要交互；H5 屏幕只做状态监控、拍照、异常标记和必要兜底。

这一点必须与参考链接保持一致：H5 MVP 的交互模型要与未来工牌一致，载体不同，Agent 体验不变。

### 2.2 任务驱动，不是工具驱动

社工首页不是空白工作台，而是“今天跑哪几家”的任务队列。系统根据排班、路线、老人服务频次、异常优先级自动组织任务，社工按顺序到达、服务、离开。

### 2.3 Agent 感，不是表单感

系统通过 Agent 主动推进流程：

- 社工端 Agent 根据 SOP、老人档案、历史记录、家属留言实时提醒。
- 管理端 Agent 查询状态、分析异常、辅助排班。
- 家属端 Agent 把服务结果转成可读的健康摘要和趋势提醒。

### 2.4 两份日志是核心交付物

每次服务结束后必须自动生成：

- 服务日志：面向站长、机构、考核，包含 SOP 完成、质量评分、异常标记、照片、备注、转写记录。
- 健康报告：面向家属，包含服务摘要、健康指标、趋势、关注提醒，不能暴露内部质检数据。

所有功能设计都围绕这两份日志的自动生成与可信交付展开。

## 3. 用户角色

| 角色 | 使用端 | 核心诉求 | 核心能力 |
| --- | --- | --- | --- |
| 一线社工 | 社工端 H5 + 蓝牙耳机 | 少操作、按任务服务、避免遗漏 SOP | 今日任务、到达确认、实时语音督导、拍照、异常标记、服务归档 |
| 站长 / 管理层 | 管理端 H5 | 排班、看状态、控质量、看异常 | 排班调度、Kanban 看板、Agent 对话、日志审核、统计分析、人员与设备管理 |
| 老人家属 | 家属端 H5 | 知道服务做了什么、老人状态如何、有问题能提醒 | 服务报告、历史趋势、月度摘要、家属留言 |
| 总部管理员 | 管理端 H5 | 跨站点管理和经营分析 | 跨站数据、权限管理、模板配置、导出报表 |

### 3.1 现场部署角色分工

| 角色 | 现场部署职责 | 必须可用的生产能力 |
| --- | --- | --- |
| 总部管理员 | 初始化机构、站点、权限、跨站模板和全局配置 | 站点管理、角色权限、全局 SOP 模板、跨站数据查看、审计 |
| 站长 / 调度管理员 | 管理本站点每日运营 | 老人/家属/社工档案、设备绑定、SOP 调试、排班、异常处理、日志审核 |
| SOP 配置员 | 把服务类型转成可执行 SOP | SOP 新建/编辑/复制/停用、步骤排序、必问项、AI 提醒话术、预期时长、试跑调试 |
| 一线社工 | 接收派单并完成上门服务 | 今日任务、路线顺序、到达确认、语音督导、拍照、异常标记、完成服务 |
| 老人家属 | 接收服务结果并补充注意事项 | 飞书报告链接、历史趋势、留言、关注提醒 |
| AI Agent | 在三端承担督导、排班和洞察 | SOP 状态机、实时转写、异常检测、报告生成、脱敏、主动提醒 |

SOP 配置员可以由站长兼任，但产品上必须是独立能力：现场上线前，必须能把“探访关爱”“助浴”等服务模板录入系统，试跑 AI 提醒话术，确认 SOP 步骤能驱动社工端服务流程。

### 3.2 从配置到上门到报告的完整现场闭环

1. 总部管理员创建机构、站点、角色权限和基础模板。
2. 站长录入社工、老人、家属、服务频次、地址、健康档案和飞书接收身份。
3. SOP 配置员创建或调整服务模板，配置步骤、必问项、AI 引导话术、预期时长和异常触发规则，并用模拟会话试跑。
4. 站长生成次日排班，Agent 给出路线和人员建议，站长确认后下发到社工端。
5. 社工打开 H5，看到今天被安排的上门服务任务、路线顺序和上次异常提示。
6. 社工到达老人家后确认到达，系统加载老人档案、SOP、历史记录和家属留言。
7. 服务过程中，AI 通过耳机实时督导，遗漏 SOP 步骤或发现风险时提醒；社工可以拍照和手动标记异常。
8. 服务结束后，系统生成内部服务日志、家属健康报告和社工 debrief。
9. 站长在管理端审核服务日志、异常标记和统计数据。
10. 家属通过飞书 channel 收到健康报告，查看趋势并留下注意事项，进入下一次服务上下文。

这条闭环必须在 MVP 中端到端可跑通，不能只交付某一端或某一个片段。

### 3.3 结果设计：Agentic Service 的大界面

本产品是 agentic service，不是传统后台系统。MVP 只做能直接跑现场的几类大界面和飞书 chat agent 入口，避免把范围扩成复杂 CRM。

#### A. 飞书 Chat Agent 入口

- **站长/调度 Agent**：在飞书里接收排班建议、异常提醒、超时提醒、日终总结；站长可以用自然语言问“今天谁还没完成”“明天怎么排班”“刘阿姨最近有什么异常”。
- **SOP 调试 Agent**：在飞书里用模拟老人档案和模拟对话试跑 SOP，返回遗漏项、提醒话术、日志字段预览；调试通过后再发布到 dashboard。
- **社工任务 Agent**：在飞书里推送今日任务、下一站提醒、异常跟进；现场服务时仍以 H5 + 耳机为主，不让社工在飞书里填表。
- **报告 Agent**：服务完成后把健康报告链接、重点关注事项、日终摘要推到对应飞书 channel。

飞书里的 Agent 必须是 chat 形态：自然语言问答 + 可点击卡片 + 确认按钮。复杂配置不放在飞书聊天里，跳转到对应业务界面完成。

#### B. 管理 Dashboard 大界面

- **今日运营台**：站长第一屏。展示今日服务总数、进行中、已完成、异常、超时、待审核；右侧固定 Agent chat 面板。
- **排班调度台**：按日期查看老人需求、社工可用性和路线建议；Agent 给出排班草案，站长确认后下发。
- **SOP 工作台**：录入探访关爱、助浴等模板；配置步骤、必问项、AI 提醒话术、预期时长、异常触发；支持模拟对话调试和发布。
- **实时服务看板**：按社工/老人/时间线切换；显示待出发、在路上、服务中、已完成、异常；异常卡片可点进详情。
- **服务详情/日志审核页**：查看转写摘要、SOP 完成、照片、录音、异常、社工备注、AI 生成的服务日志；站长审核后归档。
- **报告中心**：查看家属健康报告生成状态、飞书发送状态、失败重发、历史报告和脱敏预览。
- **档案配置页**：只保留现场必需数据：站点、社工、老人、家属、设备、飞书接收身份。

Dashboard 的 Agent 是业务界面右侧 chat panel，不单独抢主流程。Agent 给建议和解释，真正的发布、确认、审核都落在业务界面按钮上。

#### C. 社工 H5 大界面

- **今日任务页**：显示路线顺序、老人姓名、地址、服务类型、预计时长、上次异常提示。
- **服务进行页**：展示老人、服务类型、计时、录音状态、SOP 进度、最近 AI 提醒、拍照、异常标记、完成服务。
- **服务完成页**：显示 debrief、AI 摘要、可补充备注、下一站；确认后自动归档并切换任务。

社工端不做复杂 chat。现场服务时主要通过耳机听 AI 提醒，屏幕只做状态和少量操作。

#### D. 家属/报告 H5 大界面

- **单次健康报告页**：服务摘要、健康指标、关注提醒、与上次对比、社工信息。
- **历史趋势页**：时间线和关键指标趋势。
- **留言页**：家属补充注意事项，进入下一次服务上下文。

家属端不暴露内部质检、原始转写、社工评分和管理备注。

## 4. 核心用户旅程

### 4.1 社工的一天

1. 上班打开 H5，看到按路线排列的今日任务列表。
2. 到达老人家后点击“到达”，系统加载老人档案、服务类型和 SOP；未来工牌可用 GPS 自动触发。
3. 服务过程中，社工正常与老人沟通，AI 在后台实时识别对话并按 SOP 节奏提醒。
4. 遇到伤口、药瓶、环境风险等情况，社工可拍照或手动标记异常。
5. 服务结束时，社工点击“完成”，或由 AI 识别告别语自动结束。
6. 系统生成服务日志、家属健康报告和社工 debrief，并切换到下一站。
7. 全部服务完成后，系统生成日终汇总，包括服务户数、SOP 完成情况和关注标记。

### 4.2 站长的一天

1. 早上打开管理端确认排班；Agent 根据老人需求、路线、社工能力、历史异常给出建议。
2. 服务时段通过 Kanban 看板查看任务状态：待出发、在路上、服务中、已完成、异常。
3. 异常发生时，系统主动提醒，站长可点进卡片查看实时对话摘要、SOP 进度、照片和异常说明。
4. 站长可随时向 Agent 提问，例如“今天有哪些异常”“小李本周表现怎么样”“帮我排明天的班”。
5. 日终审核服务日志、异常记录和社工表现，必要时导出报表。

### 4.3 家属服务后旅程

1. 每次服务结束后，家属通过飞书 channel 收到健康报告链接。
2. 打开 H5 后看到老人姓名、服务时间、服务类型、社工姓名、服务摘要。
3. 查看血压、用药、睡眠、情绪、行动力等指标，以及与上次服务的变化。
4. 如有异常，报告中突出关注提醒和建议。
5. 家属可查看历史服务时间线和关键指标趋势。
6. 家属可留言补充注意事项，该留言进入老人档案，并在下次服务时提醒社工。

## 5. MVP 功能范围

MVP 必须一次跑通三端全流程，不分期等待硬件。

### 5.1 社工端

- 今日任务队列：按路线排序，当前任务置顶，已完成任务收起。
- 到达确认：加载老人档案、服务类型、SOP 模板和历史关注点。
- 服务进行页：展示服务计时、录音状态、SOP 进度、最近一条 AI 提示、语音波形。
- 实时语音 AI 督导：WebSocket 上行音频，ASR 转写，LLM 判断 SOP 进度和异常，TTS 通过耳机提醒。
- 拍照记录：记录伤口、药瓶、环境等证据，关联服务会话。
- 手动异常标记：在 AI 未识别或社工主动判断时补充异常。
- 服务完成页：展示 AI 生成摘要，允许社工补充语音或文字备注。
- 自动归档：确认后生成服务日志、触发家属推送，并跳转到下一任务。

### 5.2 管理端

- 现场初始化：支持站点、角色、权限、社工、老人、家属、设备、飞书接收身份的基础配置。
- 排班调度：按日排班，支持拖拽分配、Agent 自动建议、临时插入/调整/取消任务。
- Kanban 实时看板：按社工、老人、时间线三种视图查看服务任务状态。
- 异常高亮：AI 标记异常的服务卡片应高亮，并展示摘要。
- Agent 对话：支持查询异常、服务质量、社工表现、老人健康、排班建议。
- SOP 模板管理：按服务类型维护步骤、AI 引导话术、预期时长、必做检查项；MVP 预置探访关爱和助浴等模板。
- SOP 调试：支持用模拟老人档案和模拟对话试跑 SOP，查看 AI 是否按步骤提醒、是否误触发、是否能生成日志字段。
- 人员与组织管理：站点、社工、老人、家属 CRUD。
- 设备管理：手机设备 ID、蓝牙耳机型号、在线状态、最后活跃时间；未来支持切换到工牌。
- 服务日志浏览：支持按时间、社工、老人、异常状态筛选。
- 导出：支持 PDF / Excel，以满足机构和政府考核报表需求。
- 数据统计：服务量、SOP 完成率、异常率、社工效率、老人健康指标汇总。

### 5.3 家属端

- 服务报告页：移动端优先，一页式，大字体，高对比度。
- 健康指标卡片：血压、用药、睡眠、情绪、行动力等，以正常/关注/异常状态呈现。
- 服务内容摘要：用自然语言说明今天做了什么，避免内部质检话术。
- 关注提醒：异常或风险需要突出展示，并给出建议。
- 与上次对比：关键指标展示上升、下降、持平趋势。
- 历史记录：按时间倒序展示每次服务记录。
- 趋势图：血压、体重等关键指标支持时间范围切换。
- 月度健康摘要：自动汇总，为后续订阅付费能力打基础。
- 家属留言：简单文本输入，关联老人档案，进入下次服务上下文。

## 6. Agent 行为设计

### 6.1 督导 Agent

使用场景：社工端服务过程中。

输入上下文：

- 当前老人档案
- SOP 模板
- 实时对话转写
- 历史服务记录
- 家属留言
- 当前服务状态和时间

行为要求：

- 以 SOP 状态机为主线，用 LLM 判断当前步骤是否完成、是否遗漏关键问题。
- 正常对话时保持安静，只在 SOP 节点、遗漏、异常、风险时介入。
- 能识别健康指标异常、情绪异常、安全风险和服务超时。
- 语气温和、专业、像经验丰富的督导，不打断老人和社工的自然交流。

输出：

- 耳机语音提示
- 服务中页面文字提示
- SOP 进度更新
- 异常标记
- 服务后 debrief

### 6.2 管理 + 排班 Agent

使用场景：站长和管理层。

输入上下文：

- 站点服务数据
- 社工信息和能力
- 老人档案和服务频次
- 地理位置和路线
- 排班历史
- 异常和统计数据

行为要求：

- 支持自然语言查询服务状态、异常、社工表现、老人健康趋势。
- 每天生成次日排班建议，由站长确认后下发。
- 排班逻辑考虑老人服务频次、路线优化、社工能力、异常优先级、突发顶替。
- 出现超时、异常、完成率下降、排班冲突时主动推送。

### 6.3 洞察 Agent

使用场景：服务结束后的日志和报告生成。

输入：

- 完整对话转写
- SOP 完成记录
- 照片
- 老人历史档案
- 社工备注
- 家属留言

输出：

- 服务日志：给管理层，结构化、可审计、可导出。
- 健康报告：给家属，自然语言、可读、脱敏。
- 社工 debrief：一句话总结服务结果和下一站提醒。

脱敏规则：家属健康报告不能暴露内部质检评分、社工绩效评价、原始敏感对话全文和机构内部审核备注。

## 7. 技术架构

### 7.1 总体链路

三端 H5 通过 HTTPS / WebSocket 接入 API Gateway。后端服务层负责语音管道、Agent 引擎和业务服务。数据层使用 PostgreSQL、Redis 和对象存储。AI 能力优先接入阿里云百炼，接口层保留供应商切换能力。

```text
社工端 H5 / 管理端 H5 / 家属端 H5
        |
HTTPS / WebSocket
        |
API Gateway: 认证、路由、限流
        |
后端服务层
- 语音管道: ASR -> LLM -> TTS
- Agent 引擎: 督导、管理排班、洞察
- 业务服务: 调度、管理、推送、导出
        |
数据层: PostgreSQL + Redis + OSS
        |
AI 服务: Paraformer / Qwen / CosyVoice
```

### 7.2 H5 技术选型

| 模块 | 选型 |
| --- | --- |
| 前端 | React，移动端优先响应式；如后续 repo baseline 已确定其他栈，再按 baseline 调整 |
| 音频采集 | Web Audio API + getUserMedia |
| 实时通信 | WebSocket，承载音频流和状态同步 |
| 后端 | Node.js 或 Python FastAPI |
| 数据库 | PostgreSQL |
| 实时状态 | Redis |
| 文件存储 | OSS，保存录音文件、照片、导出文件 |
| 推送 | 飞书 channel |

### 7.3 AI 模型选型

主方案 follow 参考链接中的阿里云全家桶方向：

| 环节 | 主选型 | 设计理由 |
| --- | --- | --- |
| ASR | 阿里 Paraformer v2 | 流式识别、说话人分离、热词定制，适合养老护理术语 |
| LLM | Qwen 系列 | 长上下文、Function Calling、Agent 框架和 SOP 状态机结合较成熟 |
| TTS | 阿里 CosyVoice | 低首包延迟，适合耳机实时提示 |
| 实时传输 | WebSocket | H5 音频上行和提示下发都可复用 |

备选方案：

- 火山引擎全家桶：延迟优先。
- 腾讯云 ASR + 智谱 GLM：方言覆盖和中文场景优先。

接口要求：ASR、LLM、TTS 必须通过内部抽象层接入，不能把供应商 SDK 直接散落到业务逻辑里。

## 9. 权限与数据边界

| 角色 | 可见范围 | 可操作 |
| --- | --- | --- |
| 总部管理员 | 所有站点 | 全部功能、跨站分析、模板配置 |
| 站长 | 本站点 | 排班、看板、Agent 对话、日志审核、导出 |
| 社工 | 自己的任务和历史服务 | 执行服务、补充备注、拍照、标记异常 |
| 家属 | 关联老人的家属端报告 | 查看报告、趋势、留言 |

关键边界：

- 家属端只能看脱敏后的健康报告，不能看服务日志、原始转写、内部质检、社工评分。
- 社工只能查看自己的任务和相关老人必要档案，不能查看全站统计。
- 管理端导出和跨站查看需要权限审计。
- 录音、照片、健康信息属于敏感数据，必须有访问控制、操作日志和最小化展示。
- MVP 明确存储服务录音到 OSS，用于服务日志追溯、质量审核和争议处理；后续实现计划必须包含保留周期、访问审计和删除策略。

## 10. 验收标准

### 10.1 社工端验收

- 社工打开 H5 能看到当天任务队列，并按路线顺序执行。
- 到达后能加载老人档案、历史异常、SOP 模板。
- 服务过程中，AI 能根据对话推进 SOP，并在遗漏关键步骤时通过耳机提醒。
- 社工能拍照、手动标记异常、完成服务。
- 服务完成后系统能生成服务日志、健康报告、debrief，并跳到下一任务。

### 10.2 管理端验收

- 总部管理员能创建站点、角色、权限、社工、老人、家属、设备和飞书接收身份。
- SOP 配置员能录入和调试探访关爱、助浴模板，并用模拟对话验证 AI 提醒和日志字段。
- 站长能创建、调整、确认排班，并同步到社工端。
- Kanban 能实时展示任务状态和异常。
- Agent 能回答异常、社工表现、服务完成情况、排班建议类问题。
- 服务日志可以筛选、查看、导出。
- 数据统计能按站点、社工、服务类型下钻。

### 10.3 家属端验收

- 服务完成后家属能收到推送并打开 H5 报告。
- 报告能展示服务摘要、健康指标、关注提醒和与上次对比。
- 历史记录和趋势图可用。
- 家属留言能进入老人档案，并在下次服务时进入督导 Agent 上下文。

### 10.4 安全与脱敏验收

- 家属端不出现内部质检评分、社工绩效、原始敏感对话全文。
- 不同角色访问越权接口时被拒绝。
- 导出、查看敏感数据、访问照片/录音都有审计日志。

### 10.5 现场部署验收

- 从空站点开始，管理员能完成站点、人员、老人、家属、设备、SOP、排班的初始化配置。
- 站长能把一个真实服务任务派给指定社工，社工端能收到并按路线展示。
- 社工能完成一次上门服务闭环：到达、服务中 AI 督导、拍照/异常、完成、debrief。
- 系统能生成内部服务日志、家属健康报告，并通过飞书 channel 发送报告链接。
- 站长能在日终看到服务完成率、异常、SOP 完成情况和待审核日志。

## 11. MVP 非目标

- 不等待硬件工牌，不把硬件作为 MVP 前置条件。
- 不做复杂家属社交互动，家属端仅报告、趋势、留言。
- 不做完整 CRM 或财务系统。
- 不在 MVP 中追求最低 AI 成本；优先保证实时性、准确性和服务闭环。
- 不把管理端做成传统报表系统，Agent 对话和 Kanban 是主交互。

## 12. 主要风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 老人方言、噪声、多人说话导致 ASR 错误 | SOP 判断和异常识别不准 | 热词、说话人分离、低置信度兜底、人工补充备注 |
| AI 提醒过多打断服务 | 社工体验下降 | 节奏控制策略，只在 SOP 节点和异常时介入 |
| 家属报告暴露内部信息 | 隐私和信任风险 | 日志与报告分离生成，家属报告脱敏审查 |
| H5 后台录音受系统限制 | 实时督导中断 | 明确前台运行要求，服务中保持唤醒，后续工牌承接 |
| 站长不信任自动排班 | 采用阻力 | Agent 给建议和理由，站长确认后生效 |

## 13. 已确认的 MVP 决策

- 排班系统：先按金色年华没有现成排班系统处理，MVP 自建排班调度；后续如发现已有系统，再作为集成变更处理。
- 服务录音：MVP 存 OSS，不只做瞬时处理；实现计划必须覆盖保留周期、权限、审计和删除策略。
- 推送渠道：MVP 统一走飞书 channel。
- SOP 模板：MVP 至少包含探访关爱和助浴，其他模板进实现计划时再排优先级。
- 前端栈：实现计划默认按 React 规划；如后续 repo baseline 已确定其他栈，再按 baseline 调整。
