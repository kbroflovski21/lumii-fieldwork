## 1. 设计原则

Agent system 不应按传统页面模块拆分，也不应简单按岗位一一映射。正确方法是：

```text
业务用例
  -> 没有系统时由哪些人做
  -> 这些人承担什么责任
  -> 哪些责任可以被 Agent 响应、辅助、委托或自主推进
  -> 反推 Agent 责任边界
```

四类自治等级：

| 等级 | 定义 | 适用范围 |
| --- | --- | --- |
| Human-only | 必须由人完成，Agent 只能提供上下文 | 线下照护、最终责任判断、高风险外部沟通 |
| Copilot | 人发起，Agent 辅助理解、查询、总结、校验、建议 | 查手册、接线结构化、审核摘要、报告解释 |
| Delegated | 人授权目标后，Agent 执行一组任务并回报结果 | 排班草案、凭证包草案、回访任务草案、运营摘要 |
| Full autonomous | Agent 主动发现、主动创建待办或低风险产物，但必须留痕 | 到期提醒、缺证据提醒、未排需求提醒、培训提醒 |

高后果动作默认需要人工确认：

- 派单、改派、取消
- SOP 发布、规则变更
- 服务结束、异常关闭
- 审核通过、退回补充、标记结算就绪
- 凭证包导出、对外提交
- 家属报告发送
- 投诉关闭
- 权限变更和敏感导出

## 2. 责任线总览

| 责任线 | 覆盖的传统角色 | 责任焦点 | 候选 Agent 归属 |
| --- | --- | --- | --- |
| 服务规范与现场督导 | 服务主管、SOP 负责人、一线社工的规范执行部分 | SOP、培训、查手册、服务前提示、现场提醒、异常分支、服务事实记录 | 督导 Agent |
| 运营管理与调度 | 站长、调度、需求受理、人员管理 | 需求、分诊、待排班池、排班、派单、改派、服务进度、人员状态 | 管理 Agent |
| 服务凭证与合规结算 | 质检、审核、结算、站长审核部分 | 服务凭证包、缺证据、审核、结算就绪、导出、追溯 | MVP 可作为管理 Agent 的合规能力；后续可独立 |
| 家属沟通、回访与老人洞察 | 家属沟通、回访、服务主管、家属侧解释 | 家属报告、反馈分类、投诉回访、老人趋势、服务改进建议 | 洞察 Agent |
| 权限、隐私与审计护栏 | 管理员、审计、系统治理 | 访问控制、敏感操作留痕、越权拦截 | 系统护栏，不作为业务 Agent |

推荐原则：

- MVP 可见 Agent 尽量收敛为 **督导 / 管理 / 洞察** 三类。
- “服务凭证与合规结算”先作为管理 Agent 下的强边界能力，不急于变成第四个可见 Agent。
- 如果后续养护险、政府接口和结算核验成为主流程，再考虑将其升级为独立合规 Agent。
- 权限、隐私与审计是系统护栏，不应包装成普通业务 Agent。

## 3. 服务规范与现场督导责任线

### 覆盖的传统角色

- 服务主管 / SOP 负责人
- 一线社工的规范执行部分
- 质检人员对 SOP 漏项的部分判断

### 责任范围

- 服务项目和 SOP 理解。
- 社工培训和查手册。
- 服务前老人上下文提示。
- 服务中 SOP 推进、漏项提醒和异常分支。
- 服务事实记录的基础采集，例如时间线、SOP 完成、照片、录音/转写引用、社工备注。

### Copilot 能力

- 社工自然语言查手册：例如“助浴前要确认什么”。
- 服务主管生成 SOP 草案、培训说明和 FAQ。
- 服务中提醒社工当前漏了哪个必问项。
- 解释某个异常分支应该怎么处理。

### Delegated 能力

- 根据手册生成 SOP 草案。
- 运行模拟服务对话，检查 SOP 是否可执行。
- 自动整理服务过程记录草稿。
- 自动关联照片、录音、转写和 SOP 完成情况。

### Full autonomous 能力

- 新 SOP 或新规则发布后，主动提醒相关社工学习。
- 服务前主动推送老人注意事项、上次异常和家属留言。
- 服务中主动识别可能漏项、超时、风险信号。
- 服务结束后主动提示缺失凭证。

### 必须人工确认

- SOP 发布、停用、版本回滚。
- 培训合格状态。
- 服务结束。
- 异常已处理。
- 通知家属、站长或外部紧急机构，除非 SOP 明确规定强制升级。

### 关键边界

- 督导责任线拥有“事实采集”和“规范执行”责任，但不拥有结算审核的最终判断。
- 督导责任线不替代社工的线下照护、身体接触和最终现场责任。

## 4. 运营管理与调度责任线

### 覆盖的传统角色

- 站长
- 调度
- 需求受理 / 接线
- 人员管理

### 责任范围

- 内生和外生需求进入。
- 电话、小程序、社区转介、线下窗口等需求登记。
- 需求分诊、待排班池、排班、派单、改派、取消。
- 社工可用性、请假、替班、培训状态对排班的影响。
- 服务进度、异常、超时、待审核和待回访的运营推进。

### Copilot 能力

- 接线员输入一句话，结构化成服务需求。
- 提示需求还缺哪些信息。
- 查询“今天谁还没完成”“哪些服务异常”“明天还有哪些没排”。
- 解释某个老人为什么需要服务、为什么推荐某个社工。

### Delegated 能力

- 生成次日排班草案。
- 批量整理待处理异常、待审核凭证、待回访事项。
- 生成日终运营摘要。
- 为外生需求生成回拨话术和补充问题清单。

### Full autonomous 能力

- 根据固定频次、异常复访、健康风险触发内生需求待确认项。
- 主动发现明日未排需求。
- 主动发现社工请假影响已排任务。
- 主动提醒服务超时、异常未处理、投诉超时、待审核积压。

### 必须人工确认

- 外生需求进入待排班池。
- 需求关闭或标记不可服务。
- 派单、改派、取消。
- 紧急事件升级路径。
- 审核通过、退回补充、归档、导出和发送报告。

### 关键边界

- 管理责任线负责“推进业务闭环”，但不应自己解释健康趋势或改写服务事实。
- 管理责任线可以消费督导产生的服务事实，也可以消费洞察产生的趋势结论。

## 5. 服务凭证与合规结算责任线

### 覆盖的传统角色

- 质检人员
- 审核人员
- 结算人员
- 站长的凭证审核部分

### 责任范围

- 服务凭证包生成。
- 凭证完整性校验。
- 项目规则校验。
- 缺证据、缺备注、时长不足、异常未处理提示。
- 凭证审核、退回补充、结算就绪、导出。
- 对保险公司、政府、机构结算的可追溯材料准备。

### Copilot 能力

- 总结某个凭证包缺什么。
- 解释凭证为什么不满足项目规则。
- 生成审核摘要和退回补充建议。
- 帮助结算人员理解导出材料。

### Delegated 能力

- 自动生成服务凭证包草案。
- 批量检查凭证完整性。
- 批量生成待补充清单。
- 生成可导出材料草案。

### Full autonomous 能力

- 主动发现缺照片、缺备注、时长不足、异常未处理。
- 主动创建待补充队列和待审核队列。
- 主动发现凭证通过率下降或某类服务频繁缺证据。

### 必须人工确认

- 审核通过。
- 退回补充。
- 标记结算就绪。
- 凭证包导出。
- 对外提交或正式回复核验问题。

### 关键边界

- MVP 阶段建议不作为独立可见 Agent，而作为管理 Agent 下的强边界合规能力。
- 原因：MVP 不直连保险公司和政府接口，凭证包只是可导出；独立成 Agent 容易过早复杂化。
- 但它必须有清晰权限、审计、规则和确认机制，因为它直接关系到收款。

## 6. 家属沟通、回访与老人洞察责任线

### 覆盖的传统角色

- 家属沟通人员
- 回访人员
- 服务主管对老人状态和服务改进的分析部分
- 家属侧解释和反馈接收

### 责任范围

- 家属报告。
- 老人身体、情绪、行动力、用药、风险趋势。
- 家属反馈、投诉、补充信息、新需求识别。
- 异常后回访、服务后满意度回访、长期未反馈回访。
- 回访结果回流到老人档案、服务记录或新服务需求。

### Copilot 能力

- 解释家属报告中的指标和趋势。
- 把家属留言分类为普通反馈、投诉、新需求或注意事项。
- 生成回访问题清单。
- 总结老人近期状态变化。

### Delegated 能力

- 生成家属报告草案。
- 生成回访任务草案。
- 整理回访纪要。
- 把回访结果转成待确认的新服务需求或老人档案补充。

### Full autonomous 能力

- 异常服务后主动建议回访。
- 长期未反馈或高风险老人主动创建回访待办。
- 投诉超时未处理时主动提醒。
- 主动发现老人风险趋势并提示站长或家属沟通人员。

### 必须人工确认

- 家属报告发送，除非有明确自动发送规则。
- 投诉关闭。
- 向家属发送正式处理结论。
- 将家属反馈升级为服务质量问题或新服务需求。

### 关键边界

- 洞察责任线可以解释服务结果和老人趋势，但不应暴露内部服务日志、原始敏感转写、社工绩效和审核备注。
- 家属不是被替代的内部角色；面向家属时主要是解释、引导和收集信息。

## 7. 权限、隐私与审计护栏

### 覆盖的传统角色

- 总部管理员
- 系统管理员
- 审计人员

### 责任范围

- RBAC。
- 家属授权。
- 录音、照片、转写、健康信息、凭证包、导出文件和报告访问控制。
- 敏感查看、下载、导出、审核、发送、权限变更留痕。
- 越权访问和异常导出提醒。

### Agentic 能力

- Proactive：发现异常访问、频繁导出或越权尝试。
- Copilot：帮助管理员解释权限缺口。
- Delegated：生成审计摘要和敏感操作清单。

### 必须人工确认

- 权限变更。
- 敏感导出。
- 审计处理结论。

### 关键边界

- 这是系统护栏，不建议包装成普通业务 Agent。
- 所有 Agent 的读写能力都必须受这层约束。

## 8. 初步 Agent 分类建议

基于责任线和 MVP 边界，建议先采用三类可见 Agent，加一个强边界合规能力：

### 8.1 督导 Agent

责任线：

- 服务规范与现场督导。

覆盖能力：

- SOP、培训、查手册、服务前 hint、服务中提醒、异常分支、服务事实记录草稿。

不负责：

- 排班派单。
- 凭证审核通过。
- 家属报告最终发送。

### 8.2 管理 Agent

责任线：

- 运营管理与调度。
- 服务凭证与合规结算的 MVP 管理入口。

覆盖能力：

- 需求、分诊、待排班池、排班草案、派单确认、人员可用性、异常推进、待审核、凭证包导出。

不负责：

- SOP 内容权威解释。
- 老人健康趋势解释。
- 绕过确认直接做高后果动作。

### 8.3 洞察 Agent

责任线：

- 家属沟通、回访与老人洞察。

覆盖能力：

- 家属报告、老人趋势、反馈分类、投诉/回访建议、运营复盘摘要。

不负责：

- 内部凭证审核通过。
- 修改服务事实。
- 暴露内部敏感记录给家属。

### 8.4 服务凭证与合规能力

建议：

- MVP 不作为第四个可见 Agent。
- 作为管理 Agent 下的强边界能力和独立权限域。
- 如果后续直连养护险、政府接口，或凭证审核成为独立团队主流程，再升级为独立合规 Agent。

## 9. 下一步

下一步进入 agentic flow 设计。每条 flow 必须说明：

- 触发来源：人发起、Agent 主动发现、外部事件、定时任务。
- 所属责任线。
- 涉及的传统角色。
- 自治等级：Human-only、Copilot、Delegated、Full autonomous。
- 交互形态：企业聊天通道、web panel、社工 H5、家属 H5、导出文件。
- 企业聊天通道不绑定飞书卡片；复杂 dashboard、panel 和审核台通过链接或快捷入口打开外部 web 业务界面。
- 人工确认点。
- 审计和隐私要求。
