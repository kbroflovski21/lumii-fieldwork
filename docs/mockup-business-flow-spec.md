## 1. 目标

第一版 mockup 做一条完整 business flow，而不是孤立页面。

Mockup 必须是 production-ready shell：真实用户界面、真实业务状态、真实交互路径和高质量假数据；不接真实后端、不使用真实生产数据。

Production mode 默认呈现给业务用户，不展示矩阵项、flow 编号、Agent 架构解释或设计说明。Discussion mode 默认隐藏，只用于评审时标注当前界面覆盖的矩阵项、flow、权限边界和来源文档。

## 2. 覆盖范围

本 mockup 覆盖以下 PRD matrix 项：

| 矩阵项 | 角色 | Interacting Agent | 覆盖范围 | Flow |
| --- | --- | --- | --- | --- |
| #2 | 站长 / 运营 | 管理 Agent | 需求分诊、排班派单与异常调度 | F01-F04, F07, F10, F12 |
| #8 | 社工 | 督导 Agent | 社工现场任务、服务前提醒与现场 SOP | F05-F08 |
| #10 | 服务主管 / 质检 | 督导 Agent | SOP 漏项、服务日志与异常闭环 | F06-F08 |
| #11 | 服务主管 / 质检 / 审核 / 结算人员 | 管理 Agent / 凭证能力 | 服务凭证审核、导出与结算交接 | F09-F10 |

第一版不覆盖家属 H5、老人洞察、家属报告、投诉闭环和总部复盘。它们可以在后续 mockup 扩展中接入。

## 3. 角色登录

Mockup 必须支持分角色登录。登录页不是生产鉴权实现，而是静态 mockup 的角色入口，用于展示不同角色默认进入的 Agent 工作态。

| 登录角色 | 默认入口 | Agent 协作类型 | 默认可见内容 |
| --- | --- | --- | --- |
| 站长 / 运营 | 多 Agent 前置页 | 多 Agent | 管理 Agent、督导 Agent、凭证能力的重要消息和待确认事项 |
| 社工 | 督导 Agent H5 | 单 Agent | 今日任务、服务前 hint、现场 SOP、异常上报、服务记录草稿 |
| 服务主管 / 质检 | 多 Agent 前置页 | 多 Agent | 督导 Agent 的服务日志/异常闭环，管理 Agent / 凭证能力的凭证审核入口 |
| 审核 / 结算人员 | 凭证审核工作台 | 单 Agent / 强边界能力 | 凭证包、缺证据项、审核动作、导出确认和结算交接 |

多 Agent 角色登录后先进入独立前置页。前置页只展示 Agent contact list、重要消息、待确认事项和最近推动的业务任务，不承载具体业务操作。

单 Agent 角色登录后直接进入该 Agent 的工作界面。业务上下文只围绕当前任务、步骤或 Agent 产出组织。

## 4. 主业务流

第一版 business flow：

```text
管理 Agent 汇总待分诊需求
  -> 站长确认需求进入待排班池
  -> 管理 Agent 生成排班草案
  -> 站长确认派单
  -> 社工 H5 收到今日任务
  -> 督导 Agent 给服务前 hint
  -> 社工按现场 SOP 执行服务
  -> 督导 Agent 发现 SOP 漏项或异常并提示
  -> 异常升级回站长 / 服务主管
  -> 社工提交服务事实记录草稿
  -> 服务主管 / 质检查看服务记录、证据和异常闭环
  -> 管理 Agent / 凭证能力生成凭证包草案
  -> 审核 / 结算人员处理缺证据、退回或通过
  -> 人工确认后标记结算就绪或导出凭证包
```

## 5. Production UI Surfaces

### 5.1 角色登录页

用途：进入不同角色视角。

必须呈现：

- 角色列表。
- 每个角色的默认工作入口。
- 使用假数据环境的明确状态。

不得呈现：

- 技术说明。
- PRD matrix 原文。
- flow 编号。

### 5.2 多 Agent 前置页

适用角色：站长 / 运营、服务主管 / 质检。

必须呈现：

- Agent contact list：管理 Agent、督导 Agent、凭证能力。
- 每个 Agent 最近推动的任务、未读提醒、待确认数量。
- 重要消息列表；每条消息必须来自某个 Agent。
- 点击 Agent 或重要消息后进入来源 Agent 的对应工作界面，并带上业务上下文。

不得呈现：

- 功能目录。
- 独立通知中心。
- 所有业务信息的常驻铺陈。

### 5.3 管理 Agent 工作台

适用角色：站长 / 运营。

必须呈现：

- Agent chat 作为主入口。
- 当前 Agent 产出：待分诊建议、排班草案、冲突说明、异常调度建议。
- 当前任务上下文：待排班需求、社工可用性、路线冲突、老人风险、待确认动作。
- 高后果确认点：进入待排班池、派单、改派、取消、异常升级。

### 5.4 社工 H5

适用角色：社工。

必须呈现：

- 今日任务。
- 服务前 hint：老人注意事项、上次异常、服务项目凭证要求、物料提醒。
- 现场 SOP 步骤。
- 督导 Agent 的服务中提醒。
- 异常上报分支。
- 服务事实记录草稿和提交动作。

不得呈现：

- 中台报表。
- 社工绩效排名。
- 内部审核备注。

### 5.5 质检 / 凭证审核工作台

适用角色：服务主管 / 质检、审核 / 结算人员。

必须呈现：

- 服务记录摘要。
- SOP 漏项和异常闭环状态。
- 凭证包草案。
- 缺证据项和规则校验结果。
- 审核动作：退回补充、审核通过、标记结算就绪、导出凭证包。
- 导出前人工确认。

不得呈现：

- 自动对外提交给保险公司或政府。
- 未授权角色可见的原始敏感转写。

## 6. Data And State

第一版只使用假数据，但假数据必须像生产数据：

- 站点：滨江长者服务站。
- 老人：至少 3 位，包含普通任务、风险任务和异常任务。
- 社工：至少 3 位，包含可用、路线冲突、技能不匹配或请假状态。
- 服务需求：至少 5 条，覆盖电话需求、周期到期、异常复访。
- 上门任务：至少 2 条，覆盖正常服务和现场异常。
- 凭证包：至少 1 个完整、1 个缺证据、1 个异常未闭环。

状态必须贯穿页面：

- 需求从待分诊变为待排班。
- 排班草案变为已派单。
- 社工任务从待开始变为服务中，再变为待质检。
- 异常从待处理变为已升级或已闭环。
- 凭证包从草案变为待审核，再变为结算就绪或退回补充。

## 7. Discussion Mode

Discussion mode 是评审层，不是业务用户默认界面。

每个主要 surface 的 discussion panel 必须标注：

- 覆盖的 PRD matrix 项。
- 覆盖的 agentic flow。
- 当前登录角色。
- interacting agent。
- 权限边界。
- 来源文档。
- 为什么该界面是单 Agent 或多 Agent。

## 8. Staging And Deployment

Mockup artifact 放在 `deploy/current/`。

Staging 使用 [`deploy-guide.md`](deploy-guide.md) 中定义的 `http://124.221.48.52:3004/`。`3004` 是本项目 staging 端口。

第一版可以是静态 HTML/CSS/JS，不要求后端服务。页面必须在 staging 上可直接访问，并能通过角色入口体验完整 business flow。

## 9. Acceptance Criteria

- 用户可以从角色登录页进入不同角色视角。
- 站长 / 运营可以完成需求分诊、排班草案确认和派单。
- 社工可以在 H5 视角完成任务查看、SOP 执行、异常上报和服务记录提交。
- 服务主管 / 质检可以查看服务记录、SOP 漏项和异常闭环。
- 审核 / 结算人员可以查看凭证包、处理缺证据、审核通过或退回，并看到导出确认。
- 所有高后果动作都需要人工确认。
- Production mode 不展示设计说明。
- Discussion mode 默认隐藏且可以打开查看覆盖关系。
- 数据是假数据，但状态变化和业务命名像真实生产环境。
