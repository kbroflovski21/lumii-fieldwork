## 1. 方法

传统系统设计通常是：

```text
业务用例 -> 系统功能 -> 页面流程 -> 自动化点
```

Agent system 更适合按下面方式设计：

```text
业务用例 -> 现实中哪个人做这件事 -> 这个人的工作能否被同角色 Agent 接管 -> 接管程度 -> 系统能力和界面
```

本文采用四级替代程度：

| 等级 | 含义 |
| --- | --- |
| Human-only | 必须由人完成，Agent 只能提供上下文 |
| Copilot | 人发起，Agent 辅助理解、查询、总结、校验、建议 |
| Delegated | 人授权一个目标后，Agent 执行一组任务，并回报结果 |
| Full autonomous | Agent 可以主动发现、主动创建待办或低风险产物，但必须留痕 |

关键约束：

- 涉及派单、改派、取消、审核通过、凭证导出、报告发送、投诉关闭、权限变更等高后果动作，默认需要人工确认。
- Full autonomous 适合低风险、可撤销、可审计的动作，例如发现待办、生成草案、提醒缺口、创建待确认项。
- Agent 替代的是“信息处理、判断辅助、流程推进和文档生成”，不替代社工的线下照护责任。

## 2. 人类角色替代总览

| 人类角色 | 当前无系统工作 | 可替代程度 | 主要替代方向 |
| --- | --- | --- | --- |
| 机构 / 总部负责人 | 建站点、定服务项目、看质量和结算 | Copilot / Delegated | 配置建议、跨站摘要、规则缺口提醒 |
| 站长 / 调度 | 管需求、排班、派单、处理异常、审核 | Copilot / Delegated | 排班草案、冲突提醒、异常推进、审核摘要 |
| 需求受理 / 接线 | 接电话、问信息、登记需求、分诊 | Copilot / Delegated | 通话结构化、缺信息追问、需求分诊建议 |
| 服务主管 / SOP 负责人 | 培训、写 SOP、解释规范、处理服务标准问题 | Copilot / Delegated | SOP 草案、手册问答、培训提醒、模拟调试 |
| 一线社工 | 接单、上门、照护、观察、记录、处理现场问题 | Copilot | 服务前 hint、查手册、SOP 提醒、异常提示、自动记录 |
| 质检 / 审核 / 结算人员 | 查凭证、审服务、退回补充、导出材料 | Copilot / Delegated | 凭证包生成、缺证据提醒、审核摘要、导出前校验 |
| 家属沟通 / 回访人员 | 解释报告、接反馈、处理投诉、主动回访 | Copilot / Delegated | 反馈分类、回访建议、报告解释、需求回流 |
| 家属 | 看报告、提需求、投诉、补充注意事项 | Copilot | 报告问答、趋势解释、服务申请辅助 |
| 老人 | 接受服务、表达状态和需求 | Human-only | Agent 只能辅助社工理解和记录 |
| 保险公司 / 政府核验方 | 核验凭证、支付或考核 | Human-only / Copilot | MVP 只提供可导出凭证包，不替代外部核验 |

## 3. 角色替代明细

### 3.1 机构 / 总部负责人

当前人类工作：

- 决定开哪些站点、做哪些服务项目。
- 关注站点质量、服务量、投诉、凭证通过率和结算情况。
- 维护跨站服务标准和管理要求。

Copilot：

- 汇总站点运营表现。
- 解释某站点质量下降、凭证缺失或投诉升高的原因。
- 推荐需要统一培训或 SOP 调整的服务项目。

Delegated：

- 生成跨站运营周报/月报草案。
- 根据规则检查站点是否缺少配置、人员或服务项目。

Full autonomous：

- 主动发现跨站异常趋势、凭证通过率下降、投诉率异常。
- 主动创建待确认的复盘任务或培训建议。

必须人工确认：

- 新站点启用。
- 服务项目规则发布。
- 跨站权限、导出和对外提交。

### 3.2 站长 / 调度

当前人类工作：

- 查看今日和明日服务需求。
- 处理内生和外生需求。
- 排班、派单、改派、取消。
- 盯服务进度、异常、超时和社工状态。
- 审核服务结果和凭证。

Copilot：

- 解释某个老人为什么需要服务。
- 查询今日未完成、异常、超时、待审核和待回访。
- 生成排班建议、改派建议、异常处理建议。

Delegated：

- 生成次日排班草案。
- 批量整理待审核凭证。
- 生成日终运营摘要。

Full autonomous：

- 主动发现明日未排需求。
- 主动提醒社工请假影响已排任务。
- 主动发现服务超时、异常未处理、凭证缺失。

必须人工确认：

- 派单、改派、取消任务。
- 标记异常已处理。
- 审核通过、退回补充、归档。
- 导出凭证包或发送家属报告。

### 3.3 需求受理 / 接线

当前人类工作：

- 接老人、家属、社区工作人员电话。
- 询问老人是谁、发生什么、需要什么服务、希望什么时候处理。
- 判断是不是紧急、是不是符合服务范围。
- 登记需求或转给站长。

Copilot：

- 把一句自然语言转成结构化需求。
- 提示还缺哪些信息，例如老人姓名、联系方式、地址、时间偏好。
- 根据档案提示该老人服务资格、历史风险和近期服务。

Delegated：

- 为小程序或文字渠道自动生成待分诊需求。
- 根据已知信息生成回拨话术或补充问题清单。

Full autonomous：

- 对固定频次、异常复访、健康风险触发的内生需求自动创建待确认项。
- 对外生渠道的新消息主动创建待分诊项。

必须人工确认：

- 外生需求进入待排班池。
- 需求关闭或标记不可服务。
- 紧急事件升级路径。

### 3.4 服务主管 / SOP 负责人

当前人类工作：

- 编写服务标准和 SOP。
- 培训社工。
- 解答社工对服务规范的疑问。
- 处理服务过程中暴露出的 SOP 问题。
- 更新项目规则和凭证要求。

Copilot：

- 根据手册生成 SOP 草案、必问项、异常分支和凭证要求。
- 自然语言回答社工查手册问题，并引用来源。
- 分析某次服务中 SOP 漏项原因。

Delegated：

- 跑模拟服务对话，检查 SOP 是否可执行。
- 生成培训材料、FAQ 和新规则变更摘要。

Full autonomous：

- 新 SOP 或新规则发布后，主动提醒相关社工学习。
- 主动发现服务中高频漏项，创建 SOP 优化建议。

必须人工确认：

- SOP 发布、停用和版本回滚。
- 项目规则和凭证要求变更。
- 培训合格状态的最终确认。

### 3.5 一线社工

当前人类工作：

- 接收任务和路线。
- 上门与老人沟通，完成服务。
- 观察老人身体、情绪、环境和生活情况。
- 拍照、备注、处理异常。
- 服务结束后补记录。

Copilot：

- 服务前给出老人注意事项、上次异常和家属留言。
- 服务中按 SOP 提醒漏项、风险和凭证要求。
- 随时查电子手册和操作规范。
- 自动整理服务过程记录和社工 debrief。

Delegated：

- 自动生成服务记录草稿。
- 自动关联照片、录音、转写和 SOP 完成情况。

Full autonomous：

- 主动记录时间线。
- 主动识别可能异常并提醒社工确认。
- 主动提示下一站和未完成凭证。

必须人工确认：

- 现场异常是否已处理。
- 服务结束。
- 需要通知家属、站长或外部紧急机构的动作，除非 SOP 规定为强制升级。

不能替代：

- 线下照护、身体接触、环境判断的最终责任。
- 与老人建立信任的自然沟通。

### 3.6 质检 / 审核 / 结算人员

当前人类工作：

- 检查服务记录是否真实、完整、合规。
- 查照片、录音、转写、时长、异常处理和 SOP 完成。
- 退回缺材料服务。
- 汇总和导出凭证给保险公司、政府或机构结算。

Copilot：

- 汇总凭证缺口。
- 解释凭证为什么不满足项目规则。
- 生成审核摘要和退回补充建议。

Delegated：

- 自动生成服务凭证包。
- 批量检查凭证完整性。
- 生成可导出材料草案。

Full autonomous：

- 主动发现缺照片、缺备注、时长不足、异常未处理、规则不满足。
- 主动创建待补充或待审核队列。

必须人工确认：

- 审核通过。
- 退回补充。
- 标记结算就绪。
- 导出凭证包。

### 3.7 家属沟通 / 回访人员

当前人类工作：

- 给家属解释服务结果。
- 接收家属反馈、投诉、补充信息和新需求。
- 主动回访异常服务或重点老人。
- 把回访结果转给站长、社工或服务主管。

Copilot：

- 生成家属报告解释。
- 将家属留言分类为普通反馈、投诉、新需求或注意事项。
- 推荐回访问题和下一步处理。

Delegated：

- 生成回访任务。
- 整理回访纪要。
- 把回访结果回流到老人档案或服务需求池。

Full autonomous：

- 异常服务后主动建议回访。
- 长期未反馈或高风险老人主动创建回访待办。
- 投诉超时未处理时主动提醒。

必须人工确认：

- 关闭投诉。
- 向家属发送敏感解释或正式处理结论。
- 将反馈升级为服务质量问题或新服务需求。

### 3.8 家属

当前人类工作：

- 看报告。
- 理解老人状态。
- 提需求、投诉或补充注意事项。

Copilot：

- 解释报告中的指标和趋势。
- 帮助家属把自然语言补充转成注意事项或服务需求。
- 引导家属发起回访或投诉。

Delegated：

- 在授权规则内生成新的待分诊需求草稿。

Full autonomous：

- 不建议在 MVP 中让面向家属的 Agent full autonomous 处理高后果动作。

必须人工确认：

- 家属提交新需求。
- 家属投诉。
- 家属授权变更。

### 3.9 老人

当前人类工作：

- 接受服务。
- 表达身体、情绪、生活和安全状态。
- 临时提出需求或拒绝服务。

Agent 替代判断：

- 老人不是要被替代的业务角色。
- Agent 只能辅助社工理解、记录和提醒。

Copilot：

- 识别老人表达中的风险信号。
- 帮助社工记录老人状态。
- 提醒社工追问关键问题。

必须人工确认：

- 对老人状态的关键判断。
- 紧急升级。
- 服务拒绝、取消或改约。

### 3.10 保险公司 / 政府核验方

当前人类工作：

- 核验服务凭证。
- 根据规则决定支付、补贴或考核。
- 对异常凭证进行追溯。

MVP 替代判断：

- MVP 不替代保险公司或政府核验系统。
- MVP 只生成机构可导出的服务凭证包。

Copilot：

- 帮助机构解释凭证包内容。
- 检查凭证包是否满足内部配置的项目规则。

必须人工确认：

- 凭证导出。
- 对外提交。
- 对核验问题的正式回复。

## 4. 反推 Agent 分类的观察

本文不定最终 Agent 分类，但可以记录观察：

- 站长、调度、需求受理和人员管理的替代工作天然聚合为“运营/管理”类能力。
- 服务主管、SOP、培训、社工现场辅助天然聚合为“督导/规范执行”类能力。
- 质检、结算、凭证包生成和审核可能既属于管理，也可能需要独立能力边界，因为它直接关联收款和合规。
- 家属报告、老人趋势、回访和投诉处理天然聚合为“洞察/沟通/反馈”类能力。
- 老人和家属不应被当作被替代的内部工作角色；Agent 面向他们时主要是解释、引导和收集信息。

需要在 agentic flow 层继续验证的问题：

1. 质检 / 结算 / 服务凭证包是否归入管理 Agent，还是形成独立合规能力。
2. 投诉 / 回访是否归入洞察 Agent，还是归入管理 Agent 的质量闭环。
3. 培训 / 查手册是否归入督导 Agent，还是作为所有 Agent 共享的知识能力。
4. 面向家属的报告解释是否属于洞察 Agent 的对外模式，还是独立家属沟通能力。

## 5. 下一层设计输入

下一层 agentic flow 应按每个角色回答：

- Agent 何时响应人类请求。
- Agent 何时主动发现问题。
- Agent 何时生成草案。
- Agent 何时追问补信息。
- Agent 何时推荐下一步。
- Agent 何时需要人工确认。
- Agent 的输出在企业聊天通道、web 面板、H5 或导出文件中如何呈现。
- 企业聊天通道不绑定飞书卡片；复杂 dashboard、panel 和审核台通过链接或快捷入口打开外部 web 业务界面。
