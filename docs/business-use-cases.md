## 1. 业务用例写法

本文只回答真实业务里“谁要完成什么事”。每个业务用例都应明确：

- 谁是主参与者。
- 主参与者要完成什么业务动作。
- 该业务动作服务什么业务目标。
- 该业务动作产生什么业务结果。

## 2. 业务参与者

| 参与者 | 业务目标 |
| --- | --- |
| 集团管理 | 开通站点、管理服务项目、制定服务标准、查看跨站点经营和质量表现 |
| 站点运营 | 管理站点日常运营，接收服务需求，维护对象档案，排班派单，处理现场异常和回访 |
| 服务主管 | 把服务标准、SOP、培训要求和质量要求变成可执行、可指导、可核验的服务流程 |
| 社工人员 | 接收任务、上门服务、按规范执行、处理现场问题、形成服务事实记录 |
| 服务对象 | 接受服务，表达身体、情绪、生活和安全状态 |
| 服务对象家属 | 了解服务对象状态，提出需求，补充信息，投诉或要求回访 |
| 质检人员 | 审核服务记录和服务凭证，支撑养护险、政府购买服务、机构结算和质量追溯 |
| 外部结算方 | 接收或核验服务凭证，作为支付、补贴或考核依据 |

## 3. 核心业务对象

| 对象 | 说明 |
| --- | --- |
| 站点 | 服务组织、区域、人员和服务对象归属 |
| 社工人员 | 服务人员、技能、可用性、班次、培训状态 |
| 服务对象 | 基础信息、地址、身体状况、服务资格和注意事项 |
| 服务对象家属 | 联系方式、接收渠道、关注事项、反馈记录 |
| 服务项目 | 探访关爱、助浴等可执行服务类型 |
| SOP | 服务步骤、必问项、提示、异常分支、完成标准 |
| 项目规则 | 养护险、政府或机构对服务项目和凭证的要求 |
| 服务需求 | 内生或外生产生的待处理服务请求 |
| 待排班池 | 已确认、待安排社工人员和时间的服务需求集合 |
| 上门任务 | 已分配给社工人员的具体服务任务 |
| 服务过程 | 到达、服务中、异常、完成的真实过程 |
| 服务凭证 | 可证明服务真实发生和符合规范的事实材料 |
| 服务凭证包 | 支撑结算、考核、补贴和追溯的导出材料 |
| 家属报告 | 面向服务对象家属的脱敏服务结果和服务对象状态说明 |
| 反馈、投诉和回访 | 服务对象家属、服务对象或服务机构对服务结果的后续反馈和处理 |

## 4. 顶层业务用例

### 4.1 集团管理开通服务站

业务目标：让一个服务站具备开始运营的基础条件。

业务内容：

- 创建集团下属服务站、服务区域和服务范围。
- 配置人员角色、权限和站点负责人。
- 准备服务项目、基础模板和接收渠道。
- 建立飞书、企微等企业聊天通道和内部通知渠道。

### 4.2 站点运营管理社工人员

业务目标：保证服务站知道哪些社工人员可以服务、能做什么服务、什么时候可用。

业务内容：

- 维护社工人员资料、联系方式、技能和服务能力。
- 记录社工人员培训状态和上岗资格。
- 维护班次、请假、迟到、替班和临时不可用。
- 支持服务超时、突发缺人后的人员调整。

### 4.3 站点运营维护服务对象和家属档案

业务目标：明确服务对象是谁、能享受什么服务、服务对象家属如何接收信息。

业务内容：

- 维护服务对象基础信息、地址、健康情况和注意事项。
- 维护服务对象家属联系方式、接收渠道和授权关系。
- 记录服务对象是否属于养护险、政府购买服务或机构服务对象。
- 维护服务对象可服务项目、服务频次和特殊禁忌。

### 4.4 服务主管维护服务类型、SOP 和项目规则

业务目标：把服务标准变成可培训、可执行、可核验的流程。

业务内容：

- 定义探访关爱、助浴等服务项目。
- 维护 SOP 步骤、必问项、完成标准、异常分支和服务前提示。
- 维护每类服务的凭证要求，例如照片、时长、备注、录音或签名。
- 维护养护险、政府或机构对项目结算和审核的规则。

### 4.5 服务主管提供培训与知识支持

业务目标：让社工人员具备上岗能力，并在不确定时能随时查规范。

业务内容：

- 组织社工人员参加服务培训。
- 提供电子手册、服务规范、SOP 说明和常见问题。
- 支持社工人员在服务前或服务中查询操作方法、注意事项和凭证要求。
- 支持新 SOP、新政策、新项目规则的同步培训。

### 4.6 站点运营接收并分诊服务需求

业务目标：接收所有服务需求，并判断是否进入待排班池。

需求来源：

- 内生需求：固定服务频次、周期计划、上次异常复访、健康风险触发、站点运营主动安排。
- 外生需求：服务对象来电、服务对象家属来电、小程序申请、社区转介、线下窗口、临时事件。

业务内容：

- 站点运营记录外生需求。
- 判断需求是否有效、是否紧急、服务类型是什么、是否符合服务资格。
- 确认是否需要人工回访、补充信息或直接进入待排班池。

### 4.7 站点运营排班并派发上门任务

业务目标：把确认后的服务需求安排给合适社工人员和时间。

业务内容：

- 从待排班池选择服务需求。
- 根据社工人员可用性、路线、服务时长、技能、服务对象风险和优先级排班。
- 处理改约、取消、服务对象不在家、社工人员请假和临时改派。
- 将已确认任务下发给社工人员。

### 4.8 社工人员按 SOP 完成上门服务

业务目标：社工人员按规范完成服务，并在现场问题发生时知道如何处理。

业务内容：

- 社工人员接收任务，查看服务前提示、服务对象注意事项和上次异常。
- 到达服务对象家中并确认服务开始。
- 按 SOP 完成服务步骤、沟通、检查、照护和记录。
- 处理 SOP 异常分支，例如服务对象拒绝、服务对象不在家、身体异常、情绪异常、环境风险、需要通知服务对象家属或拨打 120。
- 服务结束并形成可进入凭证生成的事实材料。

### 4.9 站点运营生成并导出服务凭证包

业务目标：形成可验证、可导出的服务证明材料，支撑养护险、政府购买服务、机构结算和追溯。

业务内容：

- 采集开始/结束时间、到达证明、服务过程时间线和 SOP 完成情况。
- 关联照片、录音、转写、社工人员备注、异常处理记录和审核记录。
- 生成服务凭证包。
- 支持导出凭证包，由服务机构人工提交给外部结算方。

### 4.10 质检人员审核服务凭证

业务目标：在对外提交或内部归档前，确认凭证完整、真实、合规。

业务内容：

- 质检人员审核服务凭证，必要时请站点运营或服务主管补充说明。
- 检查是否缺照片、缺备注、缺服务时长、异常未处理或不符合项目规则。
- 决定通过、退回补充、标记问题或升级处理。
- 保留审核记录，支持后续追溯。

### 4.11 站点运营处理反馈、投诉与主动回访

业务目标：处理服务后的反馈、投诉和需要继续跟进的事项。

业务内容：

- 服务对象家属或服务对象提出投诉、质疑、补充信息或新的服务需求。
- 站点运营按需发起主动回访，例如异常后回访、服务后满意度回访、长期未反馈回访。
- 回访结果进入服务对象档案、服务记录或新的服务需求。
- 投诉和回访结果影响服务改进、社工人员管理和站点质量。

### 4.12 服务对象家属查看服务报告

业务目标：让服务对象家属了解服务内容、服务对象状态和后续关注事项。

业务内容：

- 服务对象家属收到脱敏后的服务报告，了解服务内容、服务对象状态和关注提醒。
- 服务对象家属确认报告内容是否需要补充、解释或回访。
- 服务对象家属提出新的服务需求、反馈或投诉。

### 4.13 服务主管查看服务对象洞察

业务目标：让服务主管基于服务过程和历史记录识别服务对象的持续风险，并改进服务标准和培训。

业务内容：

- 查看服务对象身体、情绪、行动力、用药和风险趋势。
- 识别需要调整 SOP、补充培训或升级处理的服务问题。
- 将复盘结果用于培训、SOP 调整和服务改进。

### 4.14 集团管理查看运营复盘

业务目标：让集团管理掌握跨站点服务表现，并推动经营、质量和资源优化。

业务内容：

- 查看服务量、完成率、异常率、凭证通过率、投诉率和社工人员效率。
- 对比不同服务站、服务项目和时间周期的运营表现。
- 将运营复盘结果用于服务项目调整、站点管理、资源投入和质量改进。

### 4.15 集团管理维护权限、隐私与留痕

业务目标：保证敏感信息只被合适角色访问，并可追溯。

业务内容：

- 不同角色只能访问自己职责范围内的数据。
- 服务对象家属不能看到内部服务日志、原始敏感对话、社工人员绩效和审核备注。
- 录音、照片、健康信息、凭证导出、报告发送和敏感查看都需要留痕。

## 5. 主业务链路

```text
站点准备
  -> 人员、服务对象、服务对象家属和服务项目准备
  -> 培训与知识支持
  -> 服务需求进入
  -> 分诊确认
  -> 待排班池
  -> 排班派单
  -> SOP 驱动的上门服务
  -> 服务凭证采集
  -> 服务凭证包生成
  -> 服务凭证审核
  -> 导出给外部结算方或用于机构结算
  -> 家属报告、投诉回访、服务对象洞察和运营复盘
```

## 6. 养老院覆盖判断

本文的业务用例可以前向覆盖养老院场景，但养老院落地时需要替换部分业务语义。

可复用部分：

- 服务对象和服务对象家属档案
- 服务项目、SOP 和项目规则
- 培训与知识支持
- 服务过程记录
- 服务凭证包
- 家属报告
- 投诉、回访、服务对象洞察和运营复盘
- 权限、隐私和留痕

需要调整部分：

- 社区上门的“路线、地址、到达服务对象家中”在养老院中替换为“楼层、房间、床位、护理单元”。
- 社区服务的“每日需求池和派单”在养老院中部分替换为“班次、巡护、护理计划和床位任务”。
- 社区场景的外生需求仍然存在，但养老院的内生需求会更强，例如护理计划、交接班、巡房、异常复查。
- 养老院需要更重的班次交接、连续护理和院内责任链，这些应在后续系统用例层补充。

结论：社区居家养老业务模型可以作为更前向的通用养老服务模型基础。本文仍以社区上门为主，但业务对象和流程命名应保留向养老院场景扩展的空间。
