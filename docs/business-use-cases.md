## 1. 设计边界

本文只回答真实业务里“谁要完成什么事”。暂不回答：

- 系统页面怎么做
- 哪个 Agent 负责
- 数据表怎么设计
- 接口怎么集成
- 具体实现优先级

MVP 业务边界：

- 服务凭证包只要求可导出，暂不直连保险公司、政府或第三方平台。
- 培训支持以培训资料、电子手册、服务规范和随时查询为主，不做完整 LMS。
- 投诉处理和主动回访是独立业务门类，不能只当作家属留言。
- 上门前提示、服务中提醒、紧急事件处理属于 SOP 驱动的上门服务流程。
- 社工请假、替班、可用性属于人员与社工管理。

## 2. 业务参与者

| 参与者 | 业务目标 |
| --- | --- |
| 机构 / 总部 | 开通站点、管理服务项目、控制质量、支撑结算和考核 |
| 站长 / 调度 | 管理站点日常运营、处理需求、排班、异常和审核 |
| 服务主管 / SOP 负责人 | 把服务标准变成可培训、可执行、可核验的流程 |
| 需求受理人员 | 接收电话、小程序、家属、老人和社区转介等外生需求 |
| 一线社工 | 接收任务、上门服务、按规范执行、处理现场问题 |
| 老人 | 接受服务，表达身体、情绪、生活和安全状态 |
| 家属 | 了解老人状态、提出需求、补充信息、投诉或要求回访 |
| 质检 / 审核 / 结算人员 | 审核服务凭证，支撑养护险、政府购买服务和机构结算 |
| 保险公司 / 政府 | 接收或核验服务凭证，作为支付、补贴或考核依据 |

## 3. 核心业务对象

| 对象 | 说明 |
| --- | --- |
| 站点 | 服务组织、区域、人员和老人归属 |
| 社工 | 服务人员、技能、可用性、班次、培训状态 |
| 老人 | 基础信息、地址、身体状况、服务资格和注意事项 |
| 家属 | 联系方式、接收渠道、关注事项、反馈记录 |
| 服务项目 | 探访关爱、助浴等可执行服务类型 |
| SOP | 服务步骤、必问项、提示、异常分支、完成标准 |
| 项目规则 | 养护险、政府或机构对服务项目和凭证的要求 |
| 服务需求 | 内生或外生产生的待处理服务请求 |
| 待排班池 | 已确认、待安排社工和时间的服务需求集合 |
| 上门任务 | 已分配给社工的具体服务任务 |
| 服务过程 | 到达、服务中、异常、完成的真实过程 |
| 服务凭证 | 可证明服务真实发生和符合规范的事实材料 |
| 服务凭证包 | 支撑结算、考核、补贴和追溯的导出材料 |
| 家属报告 | 面向家属的脱敏服务结果和老人状态说明 |
| 反馈 / 投诉 / 回访 | 家属、老人或机构对服务结果的后续反馈和处理 |

## 4. 顶层业务用例

### 4.1 机构 / 站点初始化

业务目标：让一个站点具备开始运营的基础条件。

业务内容：

- 创建机构、站点、服务区域和服务范围。
- 配置人员角色、权限和站点负责人。
- 准备服务项目、基础模板和接收渠道。
- 建立飞书、企微等企业聊天通道和内部通知渠道。

### 4.2 人员与社工管理

业务目标：保证站点知道哪些社工可以服务、能做什么服务、什么时候可用。

业务内容：

- 维护社工资料、联系方式、技能和服务能力。
- 记录社工培训状态和上岗资格。
- 维护班次、请假、迟到、替班和临时不可用。
- 支持服务超时、突发缺人后的人员调整。

### 4.3 老人 / 家属档案与服务资格管理

业务目标：明确服务对象是谁、能享受什么服务、家属如何接收信息。

业务内容：

- 维护老人基础信息、地址、健康情况和注意事项。
- 维护家属联系方式、接收渠道和授权关系。
- 记录老人是否属于养护险、政府购买服务或机构服务对象。
- 维护老人可服务项目、服务频次和特殊禁忌。

### 4.4 服务类型、SOP 与项目规则管理

业务目标：把服务标准变成可培训、可执行、可核验的流程。

业务内容：

- 定义探访关爱、助浴等服务项目。
- 维护 SOP 步骤、必问项、完成标准、异常分支和服务前提示。
- 维护每类服务的凭证要求，例如照片、时长、备注、录音或签名。
- 维护养护险、政府或机构对项目结算和审核的规则。

### 4.5 培训与知识支持

业务目标：让社工具备上岗能力，并在不确定时能随时查规范。

业务内容：

- 组织社工参加服务培训。
- 提供电子手册、服务规范、SOP 说明和常见问题。
- 支持社工在服务前或服务中查询操作方法、注意事项和凭证要求。
- 支持新 SOP、新政策、新项目规则的同步培训。

### 4.6 服务需求进入与分诊

业务目标：接收所有服务需求，并判断是否进入待排班池。

需求来源：

- 内生需求：固定服务频次、周期计划、上次异常复访、健康风险触发、站长主动安排。
- 外生需求：老人电话、家属电话、小程序申请、社区转介、线下窗口、临时事件。

业务内容：

- 需求受理人员记录外生需求。
- 判断需求是否有效、是否紧急、服务类型是什么、是否符合服务资格。
- 确认是否需要人工回访、补充信息或直接进入待排班池。

### 4.7 排班与派单

业务目标：把确认后的服务需求安排给合适社工和时间。

业务内容：

- 从待排班池选择服务需求。
- 根据社工可用性、路线、服务时长、技能、老人风险和优先级排班。
- 处理改约、取消、老人不在家、社工请假和临时改派。
- 将已确认任务下发给社工。

### 4.8 SOP 驱动的上门服务

业务目标：社工按规范完成服务，并在现场问题发生时知道如何处理。

业务内容：

- 社工接收任务，查看服务前提示、老人注意事项和上次异常。
- 到达老人家并确认服务开始。
- 按 SOP 完成服务步骤、沟通、检查、照护和记录。
- 处理 SOP 异常分支，例如老人拒绝、老人不在家、身体异常、情绪异常、环境风险、需要通知家属或拨打 120。
- 服务结束并形成可进入凭证生成的事实材料。

### 4.9 服务凭证包与导出

业务目标：形成可验证、可导出的服务证明材料，支撑养护险、政府购买服务、机构结算和追溯。

业务内容：

- 采集开始/结束时间、到达证明、服务过程时间线和 SOP 完成情况。
- 关联照片、录音、转写、社工备注、异常处理记录和审核记录。
- 生成服务凭证包。
- MVP 支持导出凭证包，由机构人工提交给保险公司、政府或其他结算方。

### 4.10 服务凭证审核

业务目标：在对外提交或内部归档前，确认凭证完整、真实、合规。

业务内容：

- 站长、主管或质检人员审核服务凭证。
- 检查是否缺照片、缺备注、缺服务时长、异常未处理或不符合项目规则。
- 决定通过、退回补充、标记问题或升级处理。
- 保留审核记录，支持后续追溯。

### 4.11 反馈、投诉与主动回访

业务目标：处理服务后的反馈、投诉和需要继续跟进的事项。

业务内容：

- 家属或老人提出投诉、质疑、补充信息或新的服务需求。
- 机构按需发起主动回访，例如异常后回访、服务后满意度回访、长期未反馈回访。
- 回访结果进入老人档案、服务记录或新的服务需求。
- 投诉和回访结果影响服务改进、社工管理和站点质量。

### 4.12 家属报告、老人洞察与运营复盘

业务目标：让不同受众看到适合自己的服务结果和趋势。

业务内容：

- 家属收到脱敏后的服务报告，了解服务内容、老人状态和关注提醒。
- 机构观察老人身体、情绪、行动力、用药和风险趋势。
- 站点和总部查看服务量、完成率、异常率、凭证通过率、投诉率和社工效率。
- 运营复盘结果用于培训、SOP 调整、排班优化和服务改进。

### 4.13 权限、隐私与留痕

业务目标：保证敏感信息只被合适角色访问，并可追溯。

业务内容：

- 不同角色只能访问自己职责范围内的数据。
- 家属不能看到内部服务日志、原始敏感对话、社工绩效和审核备注。
- 录音、照片、健康信息、凭证导出、报告发送和敏感查看都需要留痕。

## 5. 主业务链路

```text
站点准备
  -> 人员 / 老人 / 家属 / 服务项目准备
  -> 培训与知识支持
  -> 服务需求进入
  -> 分诊确认
  -> 待排班池
  -> 排班派单
  -> SOP 驱动的上门服务
  -> 服务凭证采集
  -> 服务凭证包生成
  -> 服务凭证审核
  -> 导出给保险公司 / 政府 / 机构结算
  -> 家属报告 / 投诉回访 / 老人洞察 / 运营复盘
```

## 6. 养老院覆盖判断

本文的业务用例可以前向覆盖养老院场景，但养老院落地时需要替换部分业务语义。

可复用部分：

- 老人 / 家属档案
- 服务项目、SOP 和项目规则
- 培训与知识支持
- 服务过程记录
- 服务凭证包
- 家属报告
- 投诉、回访、老人洞察和运营复盘
- 权限、隐私和留痕

需要调整部分：

- 社区上门的“路线、地址、到达老人家”在养老院中替换为“楼层、房间、床位、护理单元”。
- 社区服务的“每日需求池和派单”在养老院中部分替换为“班次、巡护、护理计划和床位任务”。
- 社区场景的外生需求仍然存在，但养老院的内生需求会更强，例如护理计划、交接班、巡房、异常复查。
- 养老院需要更重的班次交接、连续护理和院内责任链，这些应在后续系统用例层补充。

结论：社区居家养老业务模型可以作为更前向的通用养老服务模型基础。MVP 仍以社区上门为主，但不应把实体命名和流程写死到只能服务社区；系统用例层应保留向养老院的扩展口。
