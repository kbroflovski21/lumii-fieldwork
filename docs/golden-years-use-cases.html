A. 机构与站点初始化
▶ UC-B-001 初始化机构与站点 High
用例编号UC-B-001
用例名称初始化机构与站点
用例描述总部管理员创建机构信息并初始化下属服务站点，配置站点的基础属性、服务覆盖区域和管理人员
参与者主要：总部管理员
次要：--
业务目标建立机构-站点组织架构，为后续人员、老人、服务项目的挂载提供基础
前置条件总部管理员已登录管理端；机构已完成工商注册/资质审批
后置条件机构信息已创建；至少一个站点已初始化并可投入使用
触发条件新机构上线；新增服务站点；站点信息变更
基本流程1. 总部管理员进入机构管理页面
2. 录入机构基础信息：名称、统一社会信用代码、资质类型、联系人
3. 创建下属站点：站点名称、地址、所属行政区域、服务覆盖范围（街道/社区）
4. 为站点指定站长
5. 配置站点的服务能力：可提供的服务项目列表、最大社工人数
6. 系统校验信息完整性并保存
7. 站点状态置为"已就绪"，可开始录入社工和老人信息
备选流程3a. 批量导入多个站点信息 → 系统解析导入文件并逐条校验
4a. 站长尚未录入系统 → 先跳转到社工管理（UC-B-002）创建站长账号后再回来指定
异常流程E1. 站点服务覆盖区域与已有站点重叠 → 系统提醒并要求确认或调整
业务规则每个站点必须有且仅有一名站长；站点服务覆盖区域不宜重叠；机构信息创建后部分字段（如信用代码）不可修改
非功能需求--
优先级High

B. 人员与社工管理
▶ UC-B-002 管理社工档案与可用性 High
用例编号UC-B-002
用例名称管理社工档案与可用性
用例描述站长或总部管理员维护社工的完整档案信息，包括技能资质、排班班次、培训状态、请假与临时不可用，并支持替补社工的快速指定
参与者主要：站长、总部管理员
次要：--
业务目标确保社工信息完整、可用性实时准确，为排班派单提供可靠的人力资源数据
前置条件站点已初始化（UC-B-001）
后置条件社工档案已创建或更新；可用性状态已同步到排班模块
触发条件新社工入职；社工信息变更；社工请假/临时不可用；培训完成/资质到期
基本流程1. 管理员进入社工管理页面
2. 录入/编辑社工基础信息：姓名、手机号、身份证号、所属站点
3. 配置社工技能与资质：持有技能标签（探访关爱、助浴、助医陪诊等）、资质证书编号及有效期
4. 设置排班班次：默认工作时段、每日最大服务户数
5. 维护培训状态：已完成培训列表、待培训项、培训到期提醒
6. 管理可用性：当前状态（在岗/请假/培训中/临时不可用）、请假时段、不可用原因
7. 设置替补关系：为社工指定默认替补人选（技能匹配、区域就近）
8. 系统保存并同步可用性到排班模块
备选流程6a. 社工临时请假 → 系统自动检查其名下待执行任务，提示站长是否启用替补
7a. 未指定替补 → 系统根据技能和区域自动推荐替补候选
异常流程E1. 删除仍有活跃排班的社工 → 系统阻止并提示先处理排班
E2. 社工资质到期 → 系统标记为"资质过期"，排班时自动排除并提醒站长
业务规则社工必须至少持有一项技能标签才能参与排班；资质过期的社工不可分配对应类型的服务任务；总部管理员可管理所有站点社工，站长仅可管理本站点
非功能需求--
优先级High

C. 老人/家属档案与服务资格
▶ UC-B-004 管理老人档案与服务资格 High
用例编号UC-B-004
用例名称管理老人档案与服务资格
用例描述管理员维护老人的完整档案，包括基础信息、健康档案、服务资格（养护险/政府购买/机构自费）、服务频次限制和禁忌事项。服务完成后系统自动追加健康数据
参与者主要：站长、总部管理员
次要：系统（自动更新健康数据）
业务目标维护准确的老人档案和服务资格数据，支撑排班、服务执行、凭证审核和结算
前置条件站点已初始化（UC-B-001）
后置条件老人档案已创建或更新；服务资格信息可供排班和结算模块使用
触发条件老人初次签约服务；老人信息变更；服务资格变更；每次服务完成后自动更新健康数据
基本流程1. 管理员进入老人管理页面
2. 录入/编辑基础信息：姓名、性别、年龄、身份证号、地址、联系电话
3. 建立健康档案：基础病史、用药情况、身体指标基线、过敏史
4. 配置服务资格：资金来源类型（养护险/政府购买/机构自费）、对应的服务项目范围
5. 设置频次限制：每种服务类型的周/月最大服务次数（依据资格类型和项目规则）
6. 录入禁忌事项：不可提供的服务类型、特殊注意事项（如对某类清洁剂过敏）
7. 录入专属注意事项（如独居、有宠物、门锁密码等）
8. 系统保存档案
备选流程8a. 每次服务完成后，系统自动将本次采集的健康数据追加到档案中（如最新血压值、体重），不覆盖人工录入的基线数据
4a. 老人同时具备多种服务资格 → 系统支持多资格并存，排班和结算时按规则选择优先资格
异常流程E1. 服务资格到期 → 系统标记为"资格过期"，排班时提醒站长，不自动生成新需求
E2. 频次限制已满 → 系统在排班时自动拦截，提示本周期剩余可用次数
业务规则健康档案是服务执行的核心上下文来源；不同服务资格对应不同的项目规则和结算标准；禁忌事项在排班和服务启动时必须向社工提醒
非功能需求--
优先级High

后置条件家属信息已关联到老人；接收渠道已配置

2. 录入家属信息：姓名、与老人关系、联系方式
3. 配置授权关系：是否为主要联系人、是否有权查看健康报告、是否有权提交留言、是否有权发起投诉
4. 设置接收渠道偏好：微信推送/短信/小程序通知，可多选

业务目标标准化各类上门服务的流程和质量要求，明确每种服务的凭证采集标准和完成判定标准
前置条件机构已初始化（UC-B-001）

2. 创建或编辑服务项目：名称、类别、适用人群、预计时长范围
3. 编辑SOP模板：步骤列表（每步包含：步骤名、引导话术、检查项、预期时长）

5. 设定完成判定标准：必须完成的SOP步骤列表、最低服务时长、必须采集的凭证项
6. 服务主管保存模板
7. 模板生效，可在排班和服务中使用

异常流程E1. 停用正在排班中引用的模板 → 系统提醒有关联排班，确认后停用，已排班任务不受影响
E2. 删除已有服务记录引用的服务项目 → 系统阻止删除，仅允许停用
业务规则每个服务项目必须关联一个有效的SOP模板；凭证要求是审核和结算的依据；已停用的模板不影响历史记录

▶ UC-B-028 管理权限隐私与审计留痕 High
用例编号UC-B-028
用例名称管理权限隐私与审计留痕
用例描述总部管理员配置系统的角色权限体系、数据隐私保护策略和审计日志规则，确保不同角色只能访问授权范围内的数据，所有关键操作留痕可追溯

次要：--

前置条件机构已初始化（UC-B-001）；各角色人员已录入系统
后置条件权限策略已生效；隐私保护规则已启用；审计日志持续记录
触发条件系统初始化配置；新增角色或人员；权限策略调整；合规检查要求
基本流程1. 总部管理员进入权限与安全管理页面
2. 配置角色权限：
   - 定义角色（总部管理员、站长、社工、服务主管、需求受理人员、质检人员、家属）
   - 为每个角色配置数据访问范围（如站长仅可访问本站数据）
   - 为每个角色配置功能操作权限（如社工不可审核凭证）

   - 老人身份信息脱敏规则（导出时自动脱敏）

   - 健康数据的加密存储和传输要求
   - 数据保留期限和过期清理策略
4. 配置审计日志规则：
   - 需记录的操作类型（登录、数据查看、修改、导出、审核、删除等）
   - 日志保留期限
5. 系统保存配置并即时生效
6. 管理员可查看审计日志：按时间、操作人、操作类型筛选
备选流程2a. 从预置角色模板快速配置 → 管理员选择模板后微调权限
6a. 异常操作告警 → 系统检测到异常访问模式（如非工作时间大量导出数据）时自动告警
异常流程E1. 权限配置冲突（如某角色同时拥有"提交凭证"和"审核凭证"权限） → 系统提示职责分离冲突并要求调整
业务规则遵循最小权限原则；审核人员与被审核人员不可为同一人；审计日志不可删除或篡改；老人隐私数据的访问必须留痕；家属仅能查看自己授权关联的老人数据
