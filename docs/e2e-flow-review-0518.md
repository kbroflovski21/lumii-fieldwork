## 2. 已识别断点

### 断点 #1：排期 → 护理人员的同步机制（关键断点）

**问题**：站点运营在 site-operations 创建排期后，护理人员页面如何获得这些数据？

- site-operations 的排期存储在 ServiceScheduleOccurrence 实体中，有完整的 API 契约。
- careworker 页面目前使用 mock 数据，没有定义从哪个 API 拉取"我的任务"。
- 两个页面是独立的前端应用，没有共享数据层。

**影响**：这是整个流程最核心的衔接点。没有它，排期和任务就是两个孤岛。

**建议方案**：
- 定义一个 careworker-facing API：`GET /api/careworker/my-schedules?date=2026-05-19`，返回该员工当天/本周的排期列表。
- 后端从 ServiceScheduleOccurrence 表按 assignedSocialWorkerId 过滤。
- careworker 页面启动时和定期轮询拉取。
- 或者，使用 WebSocket/SSE 实现实时推送。

### 断点 #2：工牌 → 护理人员页面的真实通信

**问题**：原型使用 BroadcastChannel 模拟工牌通信。真实场景中，工牌通过蓝牙连接手机，手机上的 H5 页面如何接收工牌的状态事件？

- H5 页面在浏览器中运行，无法直接使用蓝牙 API（Web Bluetooth API 兼容性有限，iOS Safari 不支持）。
- 如果嵌入 App WebView，需要 App 提供 JSBridge 转发蓝牙事件。
- 如果工牌通过 Wi-Fi 上传，则需要服务端中转状态。

**影响**：工牌状态的实时显示和"进行中"Banner 的触发都依赖此通道。

**建议方案**：
- 短期（原型/demo）：继续用 BroadcastChannel 模拟。
- 中期（嵌入 App）：App 通过 JSBridge 向 WebView 注入工牌事件。
- 长期（独立方案）：工牌通过 4G/Wi-Fi 上报状态到服务端，页面通过 WebSocket 订阅。

### 断点 #3：服务记录 → 护理人员页面的回写

**问题**：后台生成服务记录后，护理人员如何在自己的页面看到它？

- 当前 careworker 页面的"查看报告"使用 mock 数据，是在创建任务时就预设好的。
- 真实场景中，报告是服务完成后后台异步生成的，可能需要几分钟。
- careworker 页面需要知道"这条排期现在有报告了"，并能展示。

**影响**：护理人员无法确认自己的服务是否被系统正确记录和评估。

**建议方案**：
- 同断点 #1 的 API，在排期数据中包含 serviceRecordId（如果已生成）。
- 定义 careworker-facing 报告 API：`GET /api/careworker/service-records/:id/report`。
- 或，报告生成后通过推送/通知告知护理人员。

### 断点 #4：护理人员补充说明 → 服务记录

**问题**：护理人员在"待补充"阶段输入的文字说明，最终去了哪里？

- careworker 页面有补充录入的输入框和提交按钮。
- site-operations 的 ServiceRecord 数据结构中没有明确的"careworker supplementary notes"字段。
- ServiceItem 有 transcript，ServiceException 有 description，但都不是护理人员主动录入的。

**影响**：补充说明可能丢失或无处落地。

**建议方案**：
- 在 ServiceRecord 中增加 `workerNotes: string` 字段。
- 定义 API：`POST /api/careworker/service-records/:id/notes`。
- 补充说明在 site-operations 服务记录详情中可见（作为护理人员自述，区别于 AI 推断的 structuredSummary）。

### 断点 #5：家属订阅绑定流程

**问题**：站点运营在服务对象详情中维护家属联系人和订阅偏好（daily/weekly/monthly）。但家属端如何从"被录入的联系人"变为"有权限查看报告的小程序用户"？

- role-ui-design 说家属通过小程序/公众号完成"身份绑定和订阅"。
- 但 Fieldwork 文档明确说家属端小程序"不在 Fieldwork 范围内"。
- 站点运营录入家属手机号 → 家属收到邀请 → 家属打开小程序完成绑定——这个流程的中间环节没有定义。

**影响**：family 角色的 F07 流程缺少入口。

**建议方案**：
- 定义一个邀请机制：站点运营录入家属手机号后，系统发送短信/微信模板消息，包含绑定链接。
- 家属打开链接 → 验证手机号 → 绑定服务对象 → 选择订阅频率。
- 或，由运营直接在系统中激活订阅，家属被动接收（无需家属主动操作）。

### 断点 #6：实时服务状态 → 站点运营

**问题**：站点运营的排期 Tab 有 `in_progress` 状态。护理人员按下工牌开始服务时，站点运营如何实时看到排期变为"进行中"？

- 工牌录音上传是事后的（服务结束后）。
- 工牌按钮按下的事件如何实时传递到站点运营的排期视图？

**影响**：如果站点运营无法实时看到"谁正在服务"，排期的 `in_progress` 状态就是摆设。

**建议方案**：
- 工牌按钮事件通过服务端实时广播（WebSocket/SSE）。
- 站点运营页面订阅排期状态变更。
- 或，接受延迟：工牌定期上报心跳（每分钟），服务端推断"正在使用"状态。

### 断点 #7：集团管理页面缺少实现

**问题**：集团管理有完整的角色定义（role-ui-design）和 demo 规格（demo-spec），但没有独立的页面实现。

- origin/main 的 site-operations 代码只覆盖站点运营角色。
- 集团管理的四个 Tab（运营总览、质量与规程、对象与家属、报告）没有对应的代码。

**影响**：集团管理者无法查看跨站点数据，F09 流程（管理查看报告并主动查询）没有 UI 承接。

**备注**：这可能是有意的优先级安排（先做站点运营和护理人员），但需要确认后续计划。

## 3. Exception 场景分支

### E1：工牌已绑定 + 无排期（突发上门）

```
护理人员临时上门 → 按工牌开始 → 页面显示"服务进行中"（无服务前提示）
→ 服务结束 → 后台 AI 推断服务对象和项目
→ 服务记录 reviewStatus = needs_review → 站点运营确认
```

**careworker 页面表现**：进行中 Banner 显示服务类型和对象为"未知"或 AI 推断值。补充录入阶段特别重要——护理人员可以主动填写"服务对象：陈阿姨，服务：助餐"。

**gap**：careworker 页面的补充录入目前只有一个文本框。对于无排期场景，可能需要结构化的补充（选择服务对象、选择服务类型），而不只是自由文本。

### E2：工牌未绑定 + 有排期

```
护理人员拿了闲置工牌出门 → 页面显示服务前提示（基于排期）
→ 排期时间到达 → Banner 持续显示但不进入"进行中"（无绑定工牌信号）
→ 排期结束时间到达 → Banner 消失，任务仍为"待服务"
→ 后台收到未绑定工牌录音 → AI 推断 → 站点运营确认归属
→ 确认后任务状态变为"已完成"
```

**gap**：确认后的状态回写到 careworker 页面的时机和方式（见断点 #3）。

### E3：服务中工牌异常（断电/断连）

```
录音进行中 → 工牌断电或蓝牙断连
→ careworker 页面 Header 显示"工牌离线"
→ Banner 从"服务进行中"变为"服务已完成"（自动触发 post_service）
→ 但实际服务可能还在继续
```

**问题**：工牌异常不等于服务结束。当前设计中工牌离线自动触发 post_service 可能不合理。

**建议**：工牌离线时，Banner 应显示"工牌连接中断"警告状态（橙色），而非直接进入"服务已完成"。给护理人员一个选择：重新连接或手动确认服务结束。

### E4：排期调整后护理人员未刷新

```
站点运营在排期 Tab 调整了时间或更换了人员
→ 护理人员页面仍显示旧排期
→ 护理人员按旧时间上门
```

**gap**：排期变更的实时推送机制（关联断点 #1）。如果没有实时推送，至少需要页面打开时拉取最新数据，并在排期有变更时显示明显的更新提示。

### E5：同一时间段两个排期冲突

```
站点运营误排：王建国 09:00-10:00 同时有两个服务
→ 护理人员日历显示两个重叠任务
→ 服务前提示只显示最近的一个
```

**gap**：站点运营的排期创建是否有冲突检测？careworker 页面是否需要显示冲突警告？

### E6：家属提交反馈/投诉

```
家属在小程序提交："上次服务人员态度不好"
→ 反馈进入站点运营"家属反馈"队列
→ 站点运营处理（了解情况、与护理人员沟通、回复家属）
→ 处理结果反馈给家属
```

**gap**：
- 站点运营的"家属反馈" Tab 在 role-ui-design 中有定义，但 site-operations specs 中没有这个 Tab 的详细设计。
- 护理人员是否收到通知"有家属投诉了你的服务"？（关联通知体系问题）
- 家属反馈的处理闭环流程未定义。

## 4. 各角色页面串联 Review

### 4.1 站点运营 ↔ 护理人员

| 数据流向 | 内容 | 实现状态 |
|---------|------|---------|
| 站点运营 → 护理人员 | 排期任务（谁去哪里服务谁） | ❌ 无同步机制（断点 #1） |
| 站点运营 → 护理人员 | 服务对象上下文（禁忌、注意事项） | ❌ careworker 用 mock，需要 API |
| 站点运营 → 护理人员 | 排期变更通知 | ❌ 无推送机制 |
| 护理人员 → 站点运营 | 服务开始/结束信号 | ❌ 无实时通道（断点 #6） |
| 护理人员 → 站点运营 | 补充说明 | ❌ 无落地字段（断点 #4） |
| 工牌 → 站点运营 | 录音 → 服务记录 | ✅ 有完整设计（F05 + 服务记录 Tab） |
| 站点运营 → 护理人员 | 服务记录/报告 | ❌ 无回写机制（断点 #3） |

### 4.2 站点运营 ↔ 家属

| 数据流向 | 内容 | 实现状态 |
|---------|------|---------|
| 站点运营 → 家属 | 家属联系人录入和订阅管理 | ✅ 服务对象详情中有家属联系人 Tab |
| 系统 → 家属 | 脱敏服务报告推送 | ⚠️ 有设计（F06/F07），家属端不在 Fieldwork 范围 |
| 家属 → 站点运营 | 反馈/投诉/需求 | ⚠️ role-ui-design 有提及，无详细 Tab 设计 |

### 4.3 站点运营 ↔ 集团管理

| 数据流向 | 内容 | 实现状态 |
|---------|------|---------|
| 站点数据 → 集团 | 跨站点服务量、完成率、异常率汇总 | ⚠️ 有角色定义，无页面实现（断点 #7） |
| 集团 → 站点 | 整改建议、管理指令 | ❌ 未定义 |

### 4.4 护理人员 ↔ 家属

无直接交互。家属看到的是脱敏后的服务摘要，不知道具体哪个护理人员。护理人员也不直接看到家属反馈。两者通过站点运营间接连接。这个隔离设计是合理的（隐私边界）。

## 5. 优先级建议

按影响程度排序：

| 优先级 | 断点 | 理由 |
|-------|------|------|
| P0 | #1 排期同步 | 没有它，排期和任务是两个孤岛，核心流程断裂 |
| P0 | #3 服务记录回写 | 没有它，护理人员无法确认服务被正确记录 |
| P1 | #4 补充说明落地 | 字段定义简单，但关系到数据完整性 |
| P1 | #6 实时服务状态 | 站点运营无法监控"谁正在服务" |
| P2 | #2 工牌真实通信 | 原型阶段可用模拟，正式产品需要定义 |
| P2 | #5 家属绑定流程 | 家属端不在当前范围，但需要为后续预留 |
| P2 | #7 集团管理页面 | 优先级取决于业务节奏 |
