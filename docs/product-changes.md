## 变更 001：服务主管角色 Tab 重构

来源：产品评审反馈
日期：2026-05-13

### 问题

原设计中服务主管有两个 Tab：「SOP 管理」和「SOP 调整建议」。逻辑存在问题：
- 「SOP 调整建议」是被动接收后台推送的建议，不够主动
- 没有体现督导策略的独立性——督导不只是 SOP 本身，还包括督导的方式、时机和规则

### 变更内容

将服务主管的 Tab 从：
- SOP 管理
- SOP 调整建议

改为：
- **服务 SOP 管理**
- **SOP 督导策略管理**

### 服务 SOP 管理

定义：企业内部在定义服务时，要求服务遵循的 SOP。

支持的操作：

1. **新增服务 SOP**
   - 服务主管新建一种服务类型，并定义该服务的 SOP
   - 支持通过对话方式完成新建
   - AI 需要二次确认是否发布为一个服务 SOP 版本
   - 确认后 SOP 生效，社工在服务时可看到

2. **修改已有的服务 SOP**
   - 支持通过对话方式完成调整（增删步骤、修改要求）
   - AI 需要二次确认修改内容并发布新版本
   - 确认后新版本生效，旧版本归档

### SOP 督导策略管理

定义：督导策略基于 SOP，但具有独立性和灵活性，需要基于实操情况调整。

核心能力：让服务主管设定"对于这个服务的 SOP，督导要怎么做"。

督导策略可配置项（以探访关爱为例）：

1. **服务开始时**：社工确认开始服务后，做一次简要语音提示，介绍本次服务的探访人和 SOP 必做项目
2. **服务进行中**：监听服务过程，自动根据对话记录分析和 check 已完成的项目
3. **完成顺序**：无需按 SOP 顺序进行，只要从对话记录中发现社工做了就 check 对应的项目
4. **服务结束时**：当社工通过语音确认结束，或督导探测到社工和被服务人员告别，且有项目未完成时，提示社工

### 对其他角色的影响

- 社工人员：服务时看到的 SOP 和督导行为由服务主管配置驱动
- 站点运营：SOP 版本变更在服务记录中体现

## 变更 002：集团管理角色 Tab 调整

来源：产品评审反馈
日期：2026-05-13

### 变更概览

集团管理的 Tab 从：
- 运营总览
- 质量与规程
- 对象与家属
- 报告

改为：
- **运营总览**（不变）
- **质量管理**（原「质量与规程」，重构内容）
- ~~对象与家属~~（删除）
- **服务报告**（原「报告」，重构内容）

### 变更 002-1：「质量与规程」→「质量管理」

**改名原因**：「规程」侧重于制度文本本身，而集团管理关注的是服务的质量表现，不是规程条文。名称应体现管理视角。

**内容重构逻辑**：

质量管理的对象是**服务**。系统内定义了多少种服务类型（如探访关爱、助浴、用药提醒），这个 Tab 就分为几个 block 去展示。

每个服务 block 展示三项内容：

1. **SOP 完成率** — 该服务类型下所有服务记录的 SOP 完成率统计
2. **最常见问题** — 由 AI 基于服务记录和异常数据判断给出，不是人工填写
3. **近期异常记录** — 展示一条体现了最常见问题的具体任务记录，作为证据支撑

**设计意图**：让集团管理一眼看到每种服务的质量短板在哪，有数据有案例，可以直接追问或下钻。

### 变更 002-2：删除「对象与家属」Tab

**删除原因**：集团管理的核心职责是跨站点经营和质量管理，服务对象和家属的具体信息属于站点运营层面的事务。集团管理不需要在自己的界面中直接查看个体服务对象详情。如需了解，可以通过聊天追问，或在质量管理中下钻到具体案例。

### 变更 002-3：「报告」→「服务报告」

**改名原因**：「报告」过于宽泛，不清楚是什么报告。改为「服务报告」明确了这是服务相关的报告。

**内容重构逻辑**：

这个 Tab 的作用是查看所有服务报告。在 demo 中按**站点**归类展示，便于集团管理做跨站点对比和定向追踪。

**分类维度**：按站点分组（如翠苑站、三墩站、古荡站、文新站），每个站点下展示该站点的周报、月报和专题分析。

### 对 flow 和产品设计的影响

- `agentic-flows.md` 中 F09「运营和管理查看报告并主动查询」需要更新：集团管理的查询入口不再包含对象与家属 Tab，改为通过聊天追问或质量管理下钻
- `role-ui-design.md` 中集团管理的 Tab 定义需要同步更新
- 质量管理 Tab 的 AI 判断能力需要后台 Agent 支持（基于服务记录汇总生成最常见问题）

## 变更 003：站点运营角色 Tab 调整

来源：产品评审反馈
日期：2026-05-13

### 变更概览

站点运营的 Tab 从：
- 工牌设备
- 服务任务
- 服务记录
- 待确认事项
- 家属反馈

改为：
- **工牌设备**（不变）
- **待服务**（原「服务任务」，概念统一）
- **已服务**（原「服务记录」，概念统一）
- **需确认**（原「待确认事项」，概念统一）
- **家属反馈**（不变）

### 变更 003-1：统一「服务」概念

**核心逻辑**：所有数据围绕「服务」这一个实体，用状态区分阶段，而不是用不同的 Tab 名称暗示不同的数据结构。

服务的生命周期：

```
排班创建 → 待服务（待完成状态）
  → 社工上门完成 → 后台处理生成日志和报告 → 已服务（已完成状态）
  → 如存在异常（身份对不上、服务未完成、SOP 缺失等）→ 需确认（需人工确认或补全）
```

**设计原则**：
- 所有的服务都是同一种数据结构和展示方式——一个小框展示摘要信息
- 不同 Tab 只是按状态筛选同一批数据
- 服务只有状态的区别，没有结构的区别

### 变更 003-2：Tab 内容定义

**待服务**：展示所有已排班但未完成的服务。每条服务展示：社工、服务对象、服务类型、预定时间、地址、注意事项。

**已服务**：展示所有已完成的服务。每条服务展示：社工、服务对象、服务类型、完成时间、时长、SOP 完成率、状态标签。

**需确认**：展示所有存在异常需要人工确认的服务。每条服务展示：同已服务的摘要信息，加上异常原因和待确认事项。

### 变更 003-3：已完成服务的报告交互

**交互方式**：点击已完成的服务卡片，在聊天对话框中推送一份完整报告。

**报告内容**：
- AI 摘要（由 LLM 分析转写记录生成）
- 语音日志（带声源识别的转写记录）
- SOP 完成情况（逐项 check 结果）
- 提示项（异常、关注事项、健康观察）

**设计意图**：报告不是在 Tab 内展开一个详情面板，而是通过对话框推送。这和整个产品「聊天 + Tab」的交互逻辑一致——Tab 展示结构化列表，聊天承载详细内容和追问。

### 对 flow 和产品设计的影响

- `role-ui-design.md` 中站点运营的 Tab 定义需要同步更新
- 服务实体需要统一的状态机模型：pending → completed / needs_confirmation
- 点击服务卡片推送报告到聊天的交互模式，需要在 ChatLayout 中实现一个从 Tab 到聊天的消息注入机制

## 变更 004：服务主管 Tab 再次调整 — 增加服务报告策略

来源：产品评审反馈
日期：2026-05-14

### 背景

录音有两个目的：1）事后报告和 review 服务质量；2）服务过程中实时督导/提示。需要两个独立的策略设定界面。两个策略均以服务类型为维度。

### 变更内容

服务主管的 Tab 从：
- 服务 SOP 管理
- SOP 督导策略

改为：
- **服务 SOP 管理**（不变）
- **服务报告策略**（新增）
- **服务督导策略**（原「SOP 督导策略」，去掉 SOP 前缀）

### 服务报告策略

定义：基于服务录音，事后生成报告时应提取和分析哪些内容。按服务类型配置。

可配置项：

1. **服务全文日志**（默认开启）— 是否输出完整的转写日志
2. **开场白信息提取**（默认开启）— 提取社工自我介绍（我是谁、来自哪里、服务对象是谁）
3. **客户满意度提取**（默认开启）— 提取对话中涉及满意度询问或客户主动表达满意/不满意的信息
4. **自定义提取项**（不限量添加）— 例如「识别是否出现违规向客户兜售商品」

所有配置通过对话方式修改，回复「确认发布」后生效。

### 服务督导策略（改名）

原名「SOP 督导策略」，改为「服务督导策略」。去掉 SOP 前缀，因为督导不仅基于 SOP，还包括自定义规则检测等与 SOP 无关的行为。

内容不变，仍然定义实时督导行为（服务开始/过程/结束行为、检查方式、自定义规则）。

### 对其他模块的影响

- demo-spec.md 中服务主管 Tab 定义需更新
- /api/analyze 的 prompt 可按报告策略动态调整提取内容
- page-split-spec.md 中服务主管页面的读写状态需增加 reportStrategies
