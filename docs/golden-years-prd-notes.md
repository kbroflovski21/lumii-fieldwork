# 社区居家养老 AI 督导 — H5 MVP 产品设计文档

> 2026 年 5 月 · 基于金色年华合作需求

## 1. 设计原则

1. **语音优先，零操作。** 社工在服务过程中不应该操作手机。手机揣兜里、耳机戴着，AI 通过语音完成所有交互。屏幕只是辅助监控，不是主交互界面。H5 MVP 的交互模型必须与未来硬件工牌一致——载体不同，体验不变。

2. **任务驱动，不是工具驱动。** 社工看到的是"今天跑哪几家"，不是空白页面等操作。系统根据排班自动安排一切，社工只需要到达、服务、离开。

3. **Agent 感，不是表单感。** 每个端都有 Agent 在驱动交互。社工端的 Agent 是随身督导，管理端的 Agent 是智能助理，家属端的 Agent 是健康顾问。不是让人填表、翻页面、看图表——是跟 AI 同事协作。

4. **两份日志是核心交付物。** 每次服务自动产出服务日志（对内）和健康报告（对外）。这是金色年华最关注的能力，也是商业化的基础。所有设计围绕这两份日志的自动生成展开。

## 2. 角色与端

### 社工端（一线社工）
- H5 + 蓝牙耳机，语音为主
- 查看今日任务队列
- 到达确认 → 自动开始
- 语音接收 AI 实时督导
- 服务结束自动归档

### 管理端（站长 / 管理层）
- H5，Kanban + Agent 对话 + 后台管理
- 排班调度
- 实时服务看板
- Agent 对话查询分析
- SOP / 人员 / 设备管理

### 家属端（老人家属）
- H5 健康报告页，被动推送
- 收到服务后健康摘要
- 查看历史健康趋势
- 给社工留言注意事项

## 3. Use Case：社工的一天

1. **08:30 · 上班打开 H5** — 看到今日任务列表：4 户待服务，按路线排好。第一个任务自动置顶。任务卡片：老人姓名 + 地址 + 服务类型 + 上次异常提示（如有）
2. **09:00 · 到达王奶奶家** — 点"到达"按钮（未来工牌可 GPS 自动触发）。系统自动加载王奶奶档案 + 探访关爱 SOP。耳机里听到：「王奶奶上次血压 150/95 偏高，今天请先关注血压情况。」
3. **09:05 · 服务进行中** — 社工和老人正常对话，AI 在后台实时分析。手机揣兜里，不需要操作。AI 按 SOP 节奏提醒：「用药情况还没有问，要不要现在确认一下？」
4. **09:35 · 服务结束** — 社工告别，点"完成"或 AI 识别到告别语自动结束。两份日志自动生成。耳机里听到：「今天王奶奶血压 145/90，比上次略有下降。服务日志已生成。下一站：张爷爷，助浴，文三路 12-502。」
5. **09:40 · 前往下一站** — 任务队列自动切到下一位。社工出门赶路。
6. **16:00 · 全部完成** — 今日 4 户全部服务完毕。系统生成今日汇总。「今天服务 4 户，SOP 全部完成。2 个关注标记：王奶奶血压偏高、刘阿姨情绪低落。」

## 4. Use Case：站长的一天

1. **08:00 · 排班确认** — 打开管理端，确认/调整今日排班。Agent 建议：「王奶奶上次血压异常，建议今天优先安排小李去。」排班界面：左边是社工列表，右边是待服务老人。拖拽或 Agent 自动排。
2. **09:00–16:00 · 实时监控** — Kanban 看板实时更新：谁在服务中、谁在路上、谁完成了、谁有异常。收到推送：「小李在张爷爷家的助浴服务中，AI 检测到皮肤异常标记。」
3. **随时 · Agent 对话** — 跟 Agent 问任何管理问题。「小李现在情况怎么样？」→ Agent 给实时摘要和建议。「这周哪些社工 SOP 完成率低？」→ Agent 汇总分析。
4. **17:00 · 日终审核** — 看今日汇总：服务完成率、异常标记、社工表现。审核需要特别关注的日志。

## 5. Use Case：家属

- **每次服务后** — 收到微信/短信推送：「妈妈今天的服务报告已生成。」点击链接打开 H5 页面。
  - 服务概要：探访关爱，时长 30 分钟，社工小李
  - 健康指标：血压 145/90、用药正常、情绪良好、行动力正常
  - 关注提醒：「血压仍偏高，建议持续关注，必要时就医复查」
  - 历史对比：「较上次血压下降 5mmHg，整体趋势向好」
- **查看历史** — 时间线视图：按次查看所有服务记录。关键指标折线图（血压、体重等）。月度健康摘要。
- **留言** — 给社工留注意事项：「妈妈最近腿疼，麻烦多关注」。下次服务时 AI 会提醒社工。

## 6. 社工端设计

### 页面结构

**任务列表页（默认首页）**
- 今日任务队列，按路线顺序排列
- 每张任务卡片：老人姓名 · 地址 · 服务类型 · 预计时长
- 卡片状态标记：待出发 / 已到达 / 服务中 / 已完成
- 有上次异常的卡片显示提示标签
- 当前任务高亮置顶，已完成的收起

**服务进行页**
- 顶部：老人姓名 + 服务类型 + 已用时长 + 录音状态指示
- 中部：SOP 进度条（已完成 ✓ / 进行中 → / 待完成 ○），每个步骤可展开看 AI 的引导内容
- 底部：AI 实时提示文字（最近一条 AI 说的话）+ 语音波形
- 浮动按钮：拍照（记录伤口/药瓶等）、手动标记异常、完成服务
- 整个页面设计为"可以不看"——社工揣兜也不影响服务

**服务完成页**
- AI 生成的服务摘要预览（一段话 debrief）
- 社工可以补充备注（语音输入或文字）
- 确认 → 日志归档 + 触发家属推送
- 自动跳转到下一个任务

### H5 与工牌的对应关系
- H5 的"到达"按钮 = 工牌的 GPS 自动触发
- H5 的"完成"按钮 = 工牌的告别语检测
- H5 的手机麦克风 = 工牌的双麦克风
- H5 的蓝牙耳机 = 工牌的蓝牙耳机
- 软件层完全一致，切换硬件时无需修改 Agent 逻辑

## 7. 管理端设计

### 排班调度
- 日视图：左侧社工列表 + 右侧老人列表，拖拽分配或 Agent 自动排
- 每个老人卡片显示：服务类型、地址、最近服务时间、是否有异常
- Agent 辅助：根据老人需求频次、社工地理位置、历史服务记录自动建议排班
- 支持临时插入 / 调整 / 取消任务
- 排班确认后自动下发到社工端

### Kanban 实时看板
- 按社工分列，每张卡片 = 一次服务任务
- 卡片状态：待出发 → 在路上 → 服务中 → 已完成 / 异常
- 服务中的卡片：显示已用时长、当前 SOP 进度
- 异常高亮：AI 标记了异常的卡片红色边框 + 异常摘要
- 点击卡片查看服务详情 / 实时对话文字流
- 三种视图切换：按社工 / 按老人 / 按时间线

### Agent 对话界面
管理员跟 Agent 自然语言交互，替代大部分"翻页面找数据"的操作。

### 技术管理后台

**SOP 模板管理**
- 按服务类型建模板：探访关爱、助浴、助医陪诊、送餐、康复训练等
- 每个模板包含：步骤列表、每步的 AI 引导话术、预期时长、必做检查项
- 支持新增 / 编辑 / 复制 / 启用 / 停用
- MVP 预置 2-3 个标准模板（探访关爱 + 助浴），管理员可微调

**人员与组织管理**
- 站点：名称、所属区域、站长、社工列表
- 社工：姓名、手机号、所属站点、负责老人列表、设备绑定
- 老人：姓名、地址、健康档案、服务频次和类型、家属联系方式、专属注意事项
- 家属：姓名、与老人关系、联系方式、推送渠道偏好（微信/短信）

**设备管理**
- 每个社工绑定一台手机（设备 ID）+ 一个蓝牙耳机型号
- 设备状态：在线/离线、最后活跃时间
- 后续硬件工牌到位时，同一套管理界面切换设备类型即可

**权限层级**

| 角色 | 可见范围 | 可操作 |
|------|---------|--------|
| 总部管理员 | 所有站点 | 全部功能 + 跨站分析 |
| 站长 | 本站点 | 排班 + 看板 + Agent 对话 + 日志审核 |
| 社工 | 自己的任务 | 执行服务 + 查看自己的历史 |

### 服务日志浏览
- 列表页：按时间 / 社工 / 老人 / 异常状态筛选
- 详情页：完整服务对话转写 + SOP 完成情况 + AI 异常标记 + 照片附件 + 社工备注
- 导出：PDF / Excel，对接政府考核报表需求

### 数据统计
- 服务量统计：日/周/月趋势，按站点/社工/服务类型下钻
- SOP 完成率排行
- 异常率趋势与类型分布
- 社工人均服务量与效率
- 老人健康指标汇总（管理层视角）

## 8. 家属端设计

### 服务报告页（每次服务后推送）
- 一页式设计，移动端优先，大字体，考虑中老年家属阅读习惯
- 顶部：老人姓名 + 服务日期时间 + 服务类型 + 社工姓名
- 健康指标卡片：血压、用药、睡眠、情绪、行动力，每项用颜色标示正常/关注/异常
- 服务内容摘要：自然语言描述今天做了什么（不是 SOP 检查清单，是人话）
- 关注提醒：如有异常，红色高亮 + 建议（如"血压偏高，建议近期复查"）
- 与上次对比：关键指标变化（上升/下降/持平）

### 历史记录页
- 时间线视图：每次服务一张卡片，按时间倒序
- 关键指标折线图：血压、体重等，可切换时间范围
- 月度摘要：自动生成的月度健康总结（为付费订阅打基础）

### 留言功能
- 家属可以给服务团队留言：「妈妈最近腿疼，麻烦多关注」
- 留言自动关联到老人档案，下次服务时 AI 会提醒社工
- 简单文本输入即可，不做复杂交互

## 9. Agent 行为设计

### 督导 Agent（社工端）
- 驱动模式：SOP 状态机 + LLM 判断
- 上下文：当前老人档案 + SOP 模板 + 实时对话 + 历史服务记录 + 家属留言
- 主动引导：按 SOP 步骤推进，遗漏时提醒
- 节奏控制：正常对话时保持安静，只在 SOP 节点和异常时介入
- 异常检测：健康指标异常、情绪异常、安全风险
- 语气：像经验丰富的老师傅，温和但专业

### 管理 + 排班 Agent（管理端）
- 驱动模式：自然语言对话 + 数据分析 + 调度优化
- 上下文：全站点服务数据、人员信息、老人档案、排班历史、统计
- 查询能力：服务状态、社工表现、老人健康、异常分析
- 排班调度：每天自动生成次日排班建议，站长确认即生效；调度逻辑包括老人服务频次、社工路线优化、突发顶替、异常优先级
- 对接能力：如金色年华有现成排班系统，通过 API 对接同步；没有则直接替代
- 主动推送：超时、异常、完成率下降、排班冲突时主动通知
- 输出：排班表直接下发到社工端任务队列；查询结果以自然语言回答站长
- 语气：高效、数据驱动、给结论而不是原始数据

### 洞察 Agent（日志与报告）
- 输入：完整对话转写 + SOP 记录 + 照片 + 老人历史档案
- 输出 1 — 服务日志：结构化记录，含 SOP 完成、质量评分、异常标记。给管理层
- 输出 2 — 健康报告：自然语言摘要，含指标、趋势、提醒。给家属
- 脱敏隔离：健康报告不暴露内部质检数据
- 输出 3 — 社工 debrief：一句话总结 + 下一站提醒

## 10. 技术架构与 AI 模型选型

### 整体架构

```
[社工端 H5] [管理端 H5] [家属端 H5]
       ↓ WebSocket / HTTPS ↓
      [API Gateway: 认证·路由·限流]
              ↓
         [后端服务层]
  ┌──────────┼──────────┐
  语音管道    Agent引擎   业务服务
  ASR→LLM→TTS 督导·管理·洞察 调度·管理·推送
              ↓
    [数据层: PostgreSQL · Redis · OSS]
         ↓ 脱敏后 Prompt · mTLS ↓
    [阿里云百炼 AI: Paraformer · Qwen3-Max · CosyVoice]
```

### AI 模型选型（主方案：阿里云全家桶）

| 环节 | 选型 | 说明 |
|------|------|------|
| ASR | 阿里 Paraformer v2 | CER 3.18%（国内顶尖）；WebSocket 流式；内置实时说话人分离；热词定制 |
| LLM | 阿里 Qwen3-Max | 国内综合能力最强；上下文 1M token；Function Calling + Qwen-Agent 框架 + MCP 支持 |
| TTS | 阿里 CosyVoice v3.5 | 大模型音质，flash 版首包延迟 150-300ms；声音复刻可定制 |
| 实时传输 | WebSocket 长连接 | H5 通过 getUserMedia 采集音频 |
| 端到端延迟 | ~1.0–1.5 秒 | ASR ~200ms + LLM ~500-800ms + TTS ~200-300ms |

### 备选方案

| | 方案 B：火山引擎全家桶 | 方案 C：腾讯云 + 智谱 GLM |
|---|---|---|
| ASR | Seed-ASR 2.0 | 腾讯云实时 ASR（27 种方言） |
| LLM | 豆包 Seed 2.0 Pro（首字 ~0.3s） | 智谱 GLM-5.1（754B MoE） |
| TTS | 豆包语音合成 | 腾讯云 TTS |
| 优势 | 延迟最低（~0.8-1.2s） | 方言处理最强 |
| 劣势 | ASR 无实时说话人分离 | 两家供应商，GLM 定价偏贵 |

### 单次服务成本估算（主方案）
一次 30 分钟上门服务，约 50 轮对话交互：

| 环节 | 用量 | 成本 |
|------|------|------|
| ASR（Paraformer） | 30 分钟语音 | ~0.43 元 |
| LLM（Qwen3-Max） | ~75 万 token | ~2.5 元 |
| TTS（CosyVoice） | ~5000 字 AI 输出 | ~2.0 元 |
| **合计** | | **~5 元/次** |

一个社工每天 4 户 ≈ 20 元/天。100 个社工运行一个月 ≈ 4 万元。

### H5 技术选型

| 技术 | 选型 |
|------|------|
| 前端框架 | Vue 3 或 React，移动端优先响应式 |
| 音频采集 | Web Audio API + getUserMedia |
| 实时通信 | WebSocket（音频流 + 状态同步） |
| 后端 | Node.js / Python FastAPI |
| 数据库 | PostgreSQL + Redis |
| 文件存储 | OSS |
| 推送 | 微信模板消息 / 短信 API |

## 11. UI 设计规范

**整体风格：** 绚丽背景 + Mac 风毛玻璃。柔和渐变背景、半透明毛玻璃卡片、大圆角、苹果系字体、留白充足。三端统一设计语言。

**色彩体系：**
- 页面背景：柔和浅灰 rgb(247, 249, 251)
- 卡片背景：半透明白 rgba(255, 255, 255, 0.72) + backdrop-filter: blur(20px)
- 强调色：蓝色系 #0052CC
- 文字色：深灰 #191C1E（标题），#64748B（正文），#94A3B8（辅助）
- 异常/警告：红色 #DC2626，成功/正常：绿色 #16A34A

**排版：**
- 字体栈：-apple-system, "SF Pro Text", "PingFang SC", "Noto Sans SC", system-ui, sans-serif
- 正文：14-15px，行高 1.7
- 家属端特别：正文 16-17px

**三端适配：**

| 端 | 设备 | 设计重点 |
|---|---|---|
| 社工端 | 手机竖屏 | 单列布局，操作按钮 ≥ 48px，服务中界面极简 |
| 管理端 | 电脑/平板 | 响应式多列，Kanban 横向滚动，Agent 对话框居右 |
| 家属端 | 手机竖屏 | 大字体、高对比度、一页式报告，中老年友好 |

## 12. 核心数据模型

| 实体 | 关键字段 |
|------|---------|
| 站点 Site | 名称、区域、站长（关联社工） |

| 老人 Elder | 姓名、地址、健康档案（JSON）、服务频次、默认 SOP 类型、注意事项 |
| 家属 Family | 姓名、手机号、与老人关系、推送渠道 |

| 排班 Schedule | 日期、社工、任务列表（有序） |
| 服务会话 Session | 社工、老人、SOP 模板、开始/结束时间、状态、SOP 进度、异常标记 |
| 对话记录 Transcript | 关联 Session、时间戳、说话人、文本内容 |
| 服务日志 ServiceLog | 关联 Session、SOP 完成情况、质量评分、异常列表、社工备注、照片 |
| 健康报告 HealthReport | 关联 Session + 老人、健康指标（JSON）、摘要文本、关注事项、推送状态 |

## 13. MVP 完整交付清单

MVP 一口气做完。以下所有模块都在 MVP 范围内，不分期、不预留、不等硬件。

### 社工端
- 任务队列 + 到达确认
- 实时语音 AI 督导（WebSocket 全双工）
- 服务完成 + 自动归档
- 拍照记录 + 手动标记

### 管理端
- 排班调度（管理 Agent 内置）
- Kanban 实时看板
- Agent 对话
- SOP 模板管理
- 人员与组织管理
- 设备管理
- 服务日志浏览 + 导出
- 数据统计

### 家属端
- 健康报告 H5 + 推送
- 历史记录 + 趋势图
