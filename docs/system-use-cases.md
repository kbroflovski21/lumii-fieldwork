## 1. 标注规则

| 标注 | 含义 |
| --- | --- |
| Manual | 主要由人工录入、确认或处理 |
| Assisted | 系统可以理解、补全、解释、推荐或校验 |
| Proactive | 系统应主动发现事件、风险、缺口或待办并提醒 |
| Confirm-required | 会改变业务状态、发送消息、下发任务、导出材料或归档，必须人工确认 |

## 3. 系统用例明细

### 3.1 机构 / 站点初始化

主要用户：总部管理员、站长。

系统需要支持：

- 创建机构、站点、服务区域和服务范围。
- 创建角色和权限模板。
- 配置站点负责人、服务主管、调度、社工、需求受理人员。
- 配置飞书群、通知对象和基础消息模板。
- 导入或新建基础服务项目和 SOP 模板。

输入：

- 机构信息、站点信息、人员信息、服务范围、通知渠道。

输出：

- 可运营站点。
- 初始权限和通知配置。
- 可继续配置的服务项目和 SOP 基线。

关键规则：

- 空站点不能进入排班和派单。
- 没有站长和基础服务项目时，站点不能标记为可运营。

Agentic 机会：

- Assisted：根据站点类型推荐基础角色、权限和模板。
- Proactive：发现站点缺少关键配置时提醒。

### 3.2 人员与社工管理

主要用户：站长、调度、服务主管。

系统需要支持：

- 维护社工档案、联系方式、所属站点、服务技能和服务能力。
- 维护班次、可用时段、请假、迟到、替班和临时不可用。
- 维护培训完成状态和上岗资格。
- 在排班前识别不可用社工、技能不匹配和培训未完成风险。
- 支持站长确认替班或临时改派。

输入：

- 社工资料、班次、请假信息、培训记录、技能标签。

输出：

- 可用于排班的社工池。
- 不可用、缺培训、技能不匹配提醒。
- 替班或改派确认记录。

关键规则：

- 未满足上岗资格的社工不能被自动推荐到对应服务项目。
- 请假、替班和改派会影响已排任务，必须保留变更记录。

Agentic 机会：

- Proactive：发现明日排班里有人请假或培训不满足要求。
- Assisted：推荐替班人选和改派理由。
- Confirm-required：替班、改派和人员状态变更必须人工确认。

### 3.3 老人 / 家属档案与服务资格管理

主要用户：站长、需求受理人员、服务主管。

系统需要支持：

- 维护老人基础信息、地址、联系方式、健康档案和注意事项。
- 维护家属联系人、授权关系、报告接收渠道和偏好。
- 维护老人服务资格，例如养护险、政府购买服务或机构服务对象。
- 维护可服务项目、服务频次、禁忌和历史风险。
- 在需求分诊和排班时校验资格、频次和禁忌。

输入：

- 老人档案、家属信息、资格信息、服务频次、历史服务和异常。

输出：

- 可用于分诊、排班、服务前提示和报告生成的老人上下文。
- 资格缺失、授权缺失、频次异常提醒。

关键规则：

- 家属报告只发送给授权家属。
- 服务资格和项目规则会影响能否进入待排班池和凭证包导出。

Agentic 机会：

- Assisted：从电话或备注中抽取老人关注事项。
- Proactive：发现服务频次到期、资格缺失、授权缺失或长期未服务。

### 3.4 服务类型、SOP 与项目规则管理

主要用户：服务主管、站长、总部管理员。

系统需要支持：

- 创建和维护服务项目，例如探访关爱、助浴。
- 为每个服务项目维护 SOP 步骤、必问项、服务前提示、完成标准和异常分支。
- 配置紧急事件处理分支，例如通知家属、通知站长、拨打 120。
- 配置凭证要求，例如照片、时长、备注、录音、转写或签名。
- 管理 SOP 版本，支持启用、停用、复制和发布。

输入：

- 服务项目、SOP 步骤、异常分支、凭证规则、政策规则。

输出：

- 可执行 SOP。
- 可用于服务督导、培训、凭证校验和报告生成的项目规则。

关键规则：

- 发布后的 SOP 修改必须产生版本。
- SOP 发布、停用和项目规则变更必须人工确认。

Agentic 机会：

- Assisted：根据已有手册生成 SOP 草案、必问项和凭证要求。
- Assisted：模拟服务对话，检查 SOP 是否漏项、提示是否过密。
- Confirm-required：SOP 发布和规则变更必须人工确认。

### 3.5 培训与知识支持

主要用户：社工、服务主管、站长。

系统需要支持：

- 管理培训资料、电子手册、SOP 说明、项目规则和常见问题。
- 支持社工按服务项目、老人场景、异常问题查询规范。
- 支持服务前查看当前任务相关注意事项和手册。
- 支持服务中快速查询，例如凭证怎么拍、老人拒绝怎么办。
- 记录关键培训资料的阅读或确认状态。

输入：

- 培训材料、手册、SOP、FAQ、社工查询问题。

输出：

- 查询答案、相关 SOP 片段、凭证要求、注意事项。
- 培训完成或阅读确认状态。

关键规则：

- 查询答案必须引用可追溯的手册、SOP 或项目规则。
- 服务中查询不能要求社工长时间操作屏幕。

Agentic 机会：

- Assisted：自然语言查手册。
- Proactive：新 SOP 或新规则发布后提醒相关社工学习。
- Proactive：服务前主动推送本次任务相关注意事项。

### 3.6 服务需求进入与分诊

主要用户：需求受理人员、站长、调度。

系统需要支持：

- 自动生成内生需求：固定频次、周期计划、异常复访、风险触发。
- 登记外生需求：老人电话、家属电话、小程序申请、社区转介、线下窗口、临时事件。
- 把自然语言需求结构化为老人、联系人、服务类型、时间偏好、紧急程度和备注。
- 校验服务资格、服务项目、服务频次和禁忌。
- 将有效需求放入待排班池，或标记为待补充、待回访、不可服务。

输入：

- 老人档案、服务频次、电话/小程序/人工录入需求、历史服务和异常。

输出：

- 结构化服务需求。
- 分诊结果：待排班、待补充、待回访、不可服务、紧急处理。

关键规则：

- 外生需求需要记录来源和受理人。
- 需求进入待排班池前必须有服务对象、服务类型和有效联系方式。
- 紧急事件不能只进入普通待排班池，必须触发升级规则。

Agentic 机会：

- Assisted：接线员输入一句话，系统结构化成需求单。
- Proactive：根据服务频次和风险生成内生需求。
- Assisted：建议服务类型、紧急程度和下一步处理。
- Confirm-required：需求进入待排班池、关闭或标记不可服务必须人工确认。

### 3.7 排班与派单

主要用户：站长、调度。

系统需要支持：

- 展示待排班池、社工可用性、老人地址、服务类型和优先级。
- 生成排班草案，包括社工、时间、路线、任务顺序和冲突说明。
- 支持站长调整、确认和下发任务。
- 支持取消、改约、老人不在家、请假、超时和临时改派。
- 将确认后的任务下发给社工。

输入：

- 待排班需求、社工可用性、老人地址、服务时长、服务优先级。

输出：

- 排班草案、确认排班、社工任务队列、改派记录。

关键规则：

- 排班草案不能直接改变生产排班。
- 派单、改派、取消必须人工确认。
- 改派需要保留原因和操作记录。

Agentic 机会：

- Proactive：发现待排班池有明日未排需求。
- Assisted：生成排班草案和路线理由。
- Proactive：服务超时或社工请假时提示改派。
- Confirm-required：派单、改派、取消和下发必须人工确认。

### 3.8 SOP 驱动的上门服务

主要用户：一线社工、站长、服务主管。

系统需要支持：

- 社工查看今日任务、路线顺序、服务前提示、老人注意事项和上次异常。
- 确认到达并开始服务。
- 按当前服务项目加载 SOP、必问项、异常分支和凭证要求。
- 在服务中记录时间线、SOP 完成、社工备注、照片、录音和转写。
- 支持异常分支：老人拒绝、不在家、身体异常、情绪异常、环境风险、需要通知家属或站长、需要拨打 120。
- 完成服务并进入凭证生成。

输入：

- 社工任务、老人档案、服务项目、SOP、历史异常、家属留言。

输出：

- 服务过程记录、异常记录、照片/录音/转写引用、服务完成状态。

关键规则：

- 服务中提示必须尽量少打扰社工和老人。
- 紧急事件需要按 SOP 分支升级，不能仅作为普通备注。
- 家属不可见原始敏感转写和内部处理备注。

Agentic 机会：

- Proactive：服务前提示关键注意事项。
- Assisted：根据对话推进 SOP 状态。
- Proactive：发现漏项、异常或超时时提醒。
- Confirm-required：结束服务、标记异常已处理、触发外部通知需要确认或明确规则。

### 3.9 服务凭证包与导出

主要用户：站长、质检 / 审核 / 结算人员。

系统需要支持：

- 从服务过程生成服务凭证。
- 汇总服务对象、服务人员、服务项目、开始/结束时间、SOP 完成、照片、录音/转写引用、异常处理和审核记录。
- 根据项目规则校验凭证完整性。
- 生成可导出的服务凭证包。
- 支持按老人、社工、站点、时间、服务项目导出。

输入：

- 服务过程记录、SOP 完成、照片、录音/转写、社工备注、项目规则、审核状态。

输出：

- 服务凭证包、导出文件、导出记录。

关键规则：

- MVP 不直连保险公司或政府接口，只导出凭证包。
- 凭证包导出必须有权限和审计记录。
- 缺少关键凭证时不能标记为结算就绪。

Agentic 机会：

- Assisted：自动生成凭证包草案。
- Proactive：发现缺照片、缺时长、缺备注或规则不满足。
- Confirm-required：导出和标记结算就绪必须人工确认。

### 3.10 服务凭证审核

主要用户：站长、服务主管、质检 / 审核 / 结算人员。

系统需要支持：

- 展示待审核服务凭证。
- 校验凭证完整性、服务时长、SOP 完成、异常处理和项目规则。
- 支持审核通过、退回补充、标记问题、升级处理和归档。
- 记录审核人、审核时间、审核意见和处理结果。

输入：

- 服务凭证、项目规则、异常记录、社工备注、补充材料。

输出：

- 审核结果、退回任务、归档记录、结算就绪状态。

关键规则：

- 审核动作必须留痕。
- 退回补充必须明确缺失项。
- 有未处理异常时不能直接结算就绪，除非有授权说明。

Agentic 机会：

- Assisted：总结凭证问题和补充建议。
- Proactive：日终提醒待审核、超时未审核、缺证据服务。
- Confirm-required：通过、退回、归档和结算就绪必须人工确认。

### 3.11 反馈、投诉与主动回访

主要用户：站长、需求受理人员、家属、服务主管。

系统需要支持：

- 登记家属投诉、质疑、补充信息和新的服务需求。
- 创建主动回访任务，例如异常后回访、服务后满意度回访、长期未反馈回访。
- 将回访结果写入老人档案、服务记录或新的服务需求。
- 跟踪投诉处理状态、责任人、截止时间和处理结果。
- 关联投诉与原服务凭证、社工、老人和家属报告。

输入：

- 家属反馈、老人反馈、回访任务、服务记录、投诉材料。

输出：

- 反馈记录、投诉工单、回访结果、新服务需求、服务改进建议。

关键规则：

- 投诉不能只作为普通留言，必须有状态和责任人。
- 回访可能生成新的外生需求。
- 家属反馈不能直接改写已归档凭证，只能追加记录或触发复核。

Agentic 机会：

- Assisted：从家属留言中识别投诉、需求或普通备注。
- Proactive：异常服务后自动建议回访。
- Proactive：投诉超时未处理提醒。
- Confirm-required：关闭投诉、创建新服务需求、触发回访必须人工确认。

### 3.12 家属报告、老人洞察与运营复盘

主要用户：家属、站长、总部、服务主管。

系统需要支持：

- 基于已审核或可发送的服务记录生成家属报告。
- 家属报告展示服务内容、老人状态、关注提醒和趋势，但不暴露内部审计、社工绩效、原始敏感转写。
- 汇总老人长期身体、情绪、行动力、用药和风险变化。
- 汇总站点服务量、完成率、异常率、凭证通过率、投诉率和社工效率。
- 支持运营复盘结果进入培训、SOP 调整、排班优化和服务改进。

输入：

- 服务记录、凭证审核结果、老人历史、家属反馈、运营数据。

输出：

- 家属报告、老人趋势、风险提醒、运营复盘摘要。

关键规则：

- 家属报告必须脱敏。
- 未达到发送条件的服务不能自动发给家属。
- 运营指标不能泄露给无权限角色。

Agentic 机会：

- Assisted：生成家属报告草案和老人趋势摘要。
- Proactive：发现老人风险趋势、投诉率升高、凭证通过率下降。
- Confirm-required：报告发送和敏感导出必须人工确认或符合预设自动发送规则。

### 3.13 权限、隐私与留痕

主要用户：总部管理员、站长、审计人员、系统管理员。

系统需要支持：

- 基于角色、站点、老人、任务和家属授权控制访问。
- 控制录音、照片、转写、健康信息、凭证包、导出文件和家属报告访问。
- 记录敏感数据查看、下载、导出、报告发送、审核和权限变更。
- 拒绝越权访问并返回安全错误。

输入：

- 用户身份、角色、站点、授权关系、操作请求。

输出：

- 授权结果、访问记录、审计日志、安全错误。

关键规则：

- 家属只能查看授权老人对应的脱敏报告。
- 社工只能查看自己的任务和必要老人上下文。
- 导出、下载、审核、权限变更必须留痕。

Agentic 机会：

- Proactive：发现异常访问、频繁导出或越权尝试。
- Assisted：帮助管理员解释权限缺口。
- Confirm-required：权限变更、敏感导出和审计处理必须人工确认。

## 4. 系统用例到 agentic flow 的过渡原则

下一层 agentic flow 只回答：

- 哪些系统用例由人主动发起
- 哪些系统用例由系统主动发现并提醒
- 哪些系统用例可以自然语言交互
- 哪些系统用例需要卡片或面板确认
- 哪些系统用例适合在飞书里完成，哪些必须进入 web 面板

下一层仍不应直接固化任何顶层 Agent 命名。顶层 Agent 分类应在 agentic flow 完成后再反推。
