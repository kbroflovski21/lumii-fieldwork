## 0. 参考源与防漂移约束

本文描述智能工牌驱动的最小业务闭环：工牌记录真实服务，后台自动规整服务事实，运营和管理通过聊天与结构化 Tab 查看、确认和追问。

## 1. 产品主线

```text
运营激活智能工牌
  -> 可选分配社工人员
  -> 服务主管可选设置 SOP
  -> 运营可选排单
  -> 社工人员按键开始服务记录
  -> 工牌录音并上传
  -> 后台按需检查 SOP 和异常
  -> 社工人员按键结束服务记录
  -> 后台自动生成服务记录
  -> 后台更新服务对象档案
  -> 周期性生成洞察报告
  -> 运营和管理通过聊天与 Tab 查看、确认、追问
```

核心原则：

- 工牌激活后即可使用。
- 运营可以指定社工人员，也可以让工牌动态混用。
- 服务主管可以设置 SOP；没有 SOP 时，工牌就是录音和服务记录设备。
- 运营可以排单；没有排单时，社工人员上门没有服务前提示，最终记录由录音推断生成。
- 录音事实是第一性数据。
- 结构化信息由后台自动推断，运营只处理例外。

## 2. Typical Use Cases and Flow

### F01 运营激活智能工牌

参与角色：站点运营

流程：

1. 站点运营扫描工牌设备码。
2. 手机号登录。
3. 创建或选择机构。
4. 工牌加入机构。
5. 可选指定社工人员，也可以保持动态混用。

结果：

- 工牌进入已激活状态。
- 工牌属于该机构。
- 后续服务记录归入该机构。

例外：

- 如果工牌已经被其他机构绑定，不能被新账号接管。
- 原机构释放或客服验证后，工牌才能重新绑定。

### F02 服务主管设置 SOP

参与角色：服务主管

流程：

1. 服务主管维护服务项目。
2. 服务主管设置 SOP 步骤、必问项、异常分支和完成标准。
3. SOP 生效后，后台可以在服务中检查规程执行情况。

结果：

- 有 SOP 时，后台可以检查漏项、异常和服务完整性。
- 没有 SOP 时，工牌仍可作为录音设备使用，服务结束后生成基础服务记录。

### F03 运营排单

参与角色：站点运营

流程：

1. 站点运营指定谁服务谁。
2. 输入方式可以是聊天、语音、表格、导入文件或结构化表单。
3. 系统把非结构化输入整理成任务，在结构化 Tab 中展示。

结果：

- 有排单时，社工人员上门前可以看到服务对象、地址、注意事项和服务项目。
- 没有排单时，社工人员仍可直接上门记录；后台根据录音内容推断服务对象和服务项目。

### F04 社工人员上门记录服务

参与角色：社工人员

流程：

1. 社工人员到达服务对象家中。
2. 按下工牌按钮开始记录。
3. 工牌录音并上传后台。
4. 服务结束后再次按按钮停止记录。

结果：

- 后台得到完整服务录音、开始时间、结束时间和设备身份。
- 后台根据语音内容识别社工人员、服务对象、服务项目和异常。

补充：

- 如果有 SOP，后台可以检查规程执行情况。
- 如果有输出通道，后台可以实时反馈。
- 如果没有输出通道，则默认事后规整。

### F05 后台自动规整服务记录

参与角色：后台 Agent 能力

流程：

1. 转写服务录音。
2. 判断服务对象、社工人员和服务项目。
3. 判断服务是否完整，例如语音是否中断、时长是否异常、关键环节是否缺失。
4. 如果存在 SOP，检查是否按规程执行。
5. 标出异常、风险、漏项和待确认事项。

结果：

- 生成社工服务记录档案。
- 更新服务对象档案。
- 形成待运营确认的例外事项。

### F06 后台周期性生成洞察报告

参与角色：后台 Agent 能力

流程：

1. 汇总服务记录、服务对象档案、异常、SOP 检查结果和服务趋势。
2. 周期性生成洞察报告。
3. 标出需要关注的服务对象、社工人员、站点和服务项目。

结果：

- 服务对象状态洞察。
- 社工人员规程执行洞察。
- 站点运营洞察。
- 异常和投诉趋势。

### F07 运营和管理查看报告并主动查询

参与角色：集团管理、站点运营、服务主管、质检人员

流程：

1. 用户进入聊天界面。
2. 用户直接提问，例如：
   - 最近哪些社工人员服务记录不完整？
   - 哪些服务对象最近状态变差？
   - 哪些服务没有按 SOP 做？
   - 本周异常最多的是哪个站点？
3. Agent 回答，并在 Tab 中展示结构化数据。
4. 用户确认、筛选、追问或导出。

结果：

- 用户通过聊天完成查询。
- 用户通过 Tab 查看结构化证据和明细。
- 运营只处理低置信度归属、异常和整改事项。

## 3. UI Framework

每一种角色进入系统后，都是：

```text
聊天主界面
  + 当前角色相关 Tab
```

聊天负责：

- 提问
- 查询
- 总结
- 解释
- 生成建议
- 提醒异常

Tab 负责：

- 展示结构化数据
- 展示待确认事项
- 展示记录明细
- 展示报告和趋势
- 承载确认、修改、导出等动作

红配参考只取这个结构：

- 主界面是对话。
- 结构化内容放在规整 Tab 中。
- 不使用到处漂浮的 panel。
- 不让用户先理解后台模块。

分角色 Tab 和页面设计见 [`role-ui-design.md`](role-ui-design.md)。

## 4. 自动行为

后台可以主动做这些事：

- 服务结束后自动生成服务记录。
- 服务结束后自动更新服务对象档案。
- 发现低置信度归属时提醒站点运营确认。
- 发现服务中断、时长异常或 SOP 漏项时标记异常。
- 周期性生成运营报告和服务对象洞察。
- 对需要回访、整改或复查的事项生成提醒。

自动行为不直接替人做高后果决定。涉及整改、通知、导出和对外发送时，应由对应角色确认。
